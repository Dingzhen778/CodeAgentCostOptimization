Specialized code LLMs such as CodeLlama~\cite{roziere2023code}, DeepSeekCoder~\cite{guo2024deepseek}, and Qwen2.5-Coder~\cite{hui2024qwen25codertechnicalreport} have demonstrated remarkable capabilities across diverse programming languages, significantly enhancing developer productivity. 
While early applications focused on relatively simple tasks (e.g., HumanEval and MBPP benchmarks), recent advances have enabled their use in more complex scenarios, including repository-level question answering~\cite{liu2024repoqaevaluatinglongcontext} and repository-level code completion~\cite{Yu2023CoderEvalAB,jimenez2023swe}, where models routinely process with long-context code. % (e.g., exceeding 10,000 tokens).

%长上下文压缩的必要性，简述现有方法
However, the growing context length in these tasks introduces significant computational challenges. The quadratic complexity of transformer attention mechanisms~\cite{10.5555/3295222.3295349} leads to rapidly increasing costs in both computation time and memory usage as context expands. This has created an urgent need for effective context compression techniques that can maintain model performance while reducing computational overhead.

Several approaches have been proposed for context compression, including Retrieval-Augmented Generation (RAG)~\cite{lewis2020retrieval}, LLMLingua~\cite{jiang2023llmlingua}, and AutoCompressor~\cite{chevalier2023adapting}. However, these methods were primarily designed for natural language texts and may fail to account for the unique structural and semantic characteristics of source code. While some RAG-based code completion approaches~\cite{wang2024rlcoder,zhang2023repocoder,hostnik2025retrieval,tan2024prompt} have emerged, there remains a critical gap in understanding how existing compression methods perform across different code-related tasks and programming languages.

Therefore, it has become necessary to compress long contexts.

We conduct experiments on both the Python and Java versions of the same dataset to evaluate the sensitivity of different programming languages to long context compression.

Based on the experimental results, we obtain the following findings.
\blackcircled{1} There is a clear trade-off between compression approaches: RAG-based methods (particularly fixed-window chunking) reduce generation time by up to 67\% while maintaining accuracy (EM) nearly equivalent to no compression, while hard compression techniques like LLMLingua achieve up to 10× compression ratios but suffer significant accuracy drops (30-50\% EM reduction). This demonstrates that RAG-based approaches better preserve critical code context.
\blackcircled{2} We observe a strong negative correlation between compression ratio and performance - every 10\% increase in compression (e.g., from 0.6 to 0.4 ratio) leads to an average 9.3\% drop in  accuracy (EM) across models. Notably, aggressive compression (>80\% reduction) can degrade performance below no-context baselines. %, suggesting current methods often discard structurally critical code elements.
\blackcircled{3} Model size significantly impacts compression sensitivity: while 7B models show 25-35\% EM drops at medium compression, smaller models ($\leq$1.5B) maintain stable performance ( 15-25\% EM drops). This suggests smaller models' limited context utilization makes them inherently more robust to information loss from compression.
\blackcircled{4} Java demonstrates remarkable compression resilience, with models retaining 95-100\% of original performance under moderate compression (2X), compared to Python's 80-90\%. This robustness may due  to Java's explicit type signatures and rigid syntax, which provide redundant cues that compensate for information loss during compression.

\subsection{Large Language Models for Code}
Due to advancements in the field of Natural Language Processing (NLP), general-purpose LLMs such as GPT~\cite{gpt4.1}, Gemini~\cite{gemini2.5}, Grok~\cite{grok}, Claude~\cite{claude} Qwen~\cite{hui2024qwen25codertechnicalreport} and DeepSeek~\cite{deepseek} have seen substantial progress in their performance and widespread use~\cite{brown2020language}.

To better perform on code-related tasks, a series of code specialized LLMs have been proposed. After being trained on vast and diverse datasets of source code and natural language from various domains~\cite{dai2015semi}, these models demonstrate strong code processing capabilities.

These code specialized LLMs can be categorized into two types: those that are pre-trained from scratch, and those that are continue pre-trained based on general-purpose LLMs.

%从零预训练的codebert,unixcoder,codegen,starcoder,deepseekcoder
\textbf{Train from scratch}
CodeBERT~\cite{feng2020codebert} is a bimodal pre-trained model with an encoder-decoder architecture for programming language (PL) and natural language (NL), which is capable of handling common NL-PL tasks.
UniXcoder~\cite{guo2022unixcoder} is a unified cross-modal pre-trained model for programming language. The model utilizes mask attention matrices with prefix adapters and cross-modal contents like AST and code comment to enhance code representation.
CodeGen~\cite{nijkamp2022codegen} adopts a decoder-only architecture, with a focus on multi-turn program synthesis and open-source availability.
StarCoder~\cite{li2023starcoder}, on the contrary, adopts a fill-in-the-middle objective for improved bidirectional context modeling.
DeepSeek-Coder~\cite{guo2024deepseekcoder} make the first attempt to incorporate repository-level data construction during the pre-training phase.
%基于llm的：codex,codeT5, codellama,WizardCoder,qwencoder
\textbf{Train from base}
CodeX~\cite{chen2021evaluatinglargelanguagemodels} adapts GPT architecture and is pretrained on a large corpus of GitHub code using next-token prediction. 
CodeLlama~\cite{rozière2024codellamaopenfoundation} extends LLaMA with code-specific tokenization and longer contexts. 
Qwen2.5-Coder~\cite{hui2024qwen25codertechnicalreport} is trained based on Qwen2.5. It parses code data using ASTs and employs rejection sampling for a two-stage post-training.

All these LLMs for Code significantly increase the chance that AI (i.e.,
LLMs) writes a piece of code instead of a human developer.

In this study, we specifically analyzed these code specificed LLMs to evaluate: CodeLlama (7B), DeeepSeek-Coder (6.7B) and Qwen2.5-Coder (7B),  since they are open source, perform well on many software engineering tasks, and have been widely used.

\subsection{Software Engineering Tasks with Long Context}
These models demonstrate remarkable performance in SE tasks such as code generation, code summarization, and code completion~\cite{chen2021evaluatinglargelanguagemodels, austin2021program,nijkamp2022codegen,wang2024rlcoder,zhang2023repocoder}. 
Despite increasingly longer context windows, LLMs exhibit significant limitations in long context scenarios, especially when code completion or quetion answering requires cross-function or structural dependencies~\cite{ding2023crosscodeeval,Yu2023CoderEvalAB,guo2025omnigirlmultilingualmultimodalbenchmark}. 
For example, even within code completion tasks, current LLMs have achieved a pass@1 accuracy of up to 96\% on the function-level code completion benchmark HumanEval~\cite{chen2021evaluatinglargelanguagemodels,evalplus-leaderboard}, while the best solved rate on the GitHub issue resolution benchmark Swebench barely reaches 60\%~\cite{jimenez2023swe,swebench-leaderboard}.

To address these challenges, several long code benchmarks have been introduced, 
such as CrossCodeEval~\cite{ding2023crosscodeeval} and CoderEval~\cite{Yu2023CoderEvalAB} in the area of repository-level code completion, RepoQA~\cite{liu2024repoqaevaluatinglongcontext} in the field of repository-level code question answering, and Swebench~\cite{jimenez2023swe} and OmniGIRL~\cite{guo2025omnigirlmultilingualmultimodalbenchmark} for GitHub issue resolution.

\subsection{Long Context Compression Methods}
%现有方法,自行分类,现在有旧的软硬压缩方法
Compression methods for long context inputs can be categorized into RAG-based methods, hard (prompt) compression methods and soft (prompt) compression methods.

Most long context compression methods are based on \textbf{RAG}~\cite{parvez2021retrievalaugmentedcodegeneration}. These methods achieve prompt compression by dynamically retrieving and injecting only the most relevant code blocks into the input prompt. 
RAG generally consists of three stages: preprocessing, retrieval, and generation. The main differences among various RAG-based compression methods typically lie in the retrieval stage, where the retrieval logic can vary significantly between methods.
For example, in the domain of repository-level code completion, the RepoCoder~\cite{zhang2023repocoder} method uses generated code completions to enhance the retrieval process, iterating through multiple rounds to accomplish the task. 
Another method, RLCoder~\cite{wang2024rlcoder}, also designed for repository-level code completion, leverages perplexity to train the retriever in order to select the most effective code blocks.

\textbf{Hard compression} methods directly manipulate the input by removing or rephrasing less informative content. LLMLingua~\cite{jiang2023llmlingua}, LongLLMLingua~\cite{jiang2023longllmlingua}, and Selective Context~\cite{li-etal-2023-compressing} use learned knowledge or statistical importance scores to prune uninformative tokens or sentences. 
LLMLingua-2~\cite{pan2024llmlingua} advances this by using data distillation from GPT-4 to train a token classification model, achieving efficient and faithful compression.

In contrast, \textbf{soft compression} methods~\cite{chevalier2023adapting,mu2024learning,500xcompressor} summarize input context into dense vectors or prefix embeddings. Soft compression methods typically achieve higher compression ratios than hard compression methods, but usually require training a compression model which is highly related to the generative model, making them impractical in closed-source model like GPT and Gemini. 

%上面压缩方法的问题：只为自然语言设置，引出DietCode和SLimCode
However, these methods are primarily designed for natural language, and often fail to capture the structural and semantic regularities of source code, leading to suboptimal performance in code-related tasks. This has led to a growing body of research focused on code-specific compression. ShortenDoc targets docstring compression specifically, whereas
our method targets source code, which typically dominates the input in long context scenarios. DietCode~\cite{zhang2022diet} is a code compression framework aimed at code understanding tasks, mainly targets Python and Java programs. It leverages abstract syntax trees (AST) and reduces input code length through syntax-aware and semantic-preserving strategies, but its reliance on model-specific
attention reduces adaptability across different architectures. SlimCode~\cite{wang2024naturalbestmodelagnosticcode} also adopts structured pruning based on AST. However, compared to DietCode, SlimCode places greater emphasis on code executability and compilability. It attempts to preserve the program structure as much as possible, ensuring that the compressed code remains verifiable by the compilers.

\subsection{Datasets}

To evaluate the performance of long-context compression methods in the code domain, we select the repository-level code completion task for our experiments. In this task, the code context typically consists of extensive code background information, and the specific objective is to complete a function or a class, with the ground truth being the actual implementation of the function or class. To objectively assess model performance, we conduct experiments on the following three dataset, CrossCodeEval~\cite{ding2023crosscodeeval} , RepoEval~\cite{zhang2023repocoder} and LCC~\cite{guo2023longcoder}:

%RepoQA~\cite{liu2024repoqaevaluatinglongcontext}.

\underline{CrossCodeEval} has 9928 code completion instances in four programming languages: Python, Java, TypeScript, and C\#. The data is collected from 99 code repositories created from March 5, 2023 to June 15, 2023, each with at least 3 stars and between 10 and 50 files. We select a subset of benchmark in the Python programming language for our experiments, comprising a total of 2,665 samples.
The quality of the generated code is evaluated by directly comparing it with the reference code using exact match (EM) and edit similarity (ES) metrics. 


\underline{RepoEval} selects a set of Python repositories created after January 1, 2022, each with more than 100 stars, to construct three sub-datasets at different levels of granularity: line completion level, API invocation completion level, and function body completion level. 
In total, these sub-datasets contain 3,573 samples. RepoEval provides evaluation at multiple granularities, as well as more precise assessments based on unit tests.

% \underline{RepoQA} is a collection of 500 code search tasks from 50 popular repositories across 5 modern programming languages (Python, C++, Java, TypeScript, and Rust) with at least 100 GitHub stars. Researchers build an initial task, namely Searching Needle Function (SNF), which exercises LLMs to search functions given their natural-language description, i.e., LLMs cannot find the desired function if they cannot understand the description and code.

\underline{LCC} targets the code
completion task under long context of relevant functions. The LCC dataset includes three programming languages: Python, Java, and C\#. Each language has a test set of 10K samples. The sequence length in this dataset ranges from 512 to 10,000 tokens, with an average length of around 2,000 tokens.

\subsection{Evaluation Metrics}
The evaluation of compression methods encompasses two primary dimensions: compression efficiency and downstream generation performance with compressed context. 
We use \textbf{compression ratio} to evaluate compression efficiency:
\begin{equation}
Compression Ratio = \frac{\lvert{Context_{compressed}}\rvert}{\lvert{Context_{original}}\rvert}
\end{equation}
where $Context_{compressed}$ represents the length of the context input to the model after compression,  
while $Context_{original}$ represents the length of the context before compression.

For the code completion task, we follow standard practice~\cite{Yu2023CoderEvalAB,ding2023crosscodeeval} and use \textbf{EM} and \textbf{ES} to evaluate the performance of the models.

\textbf{EM (Exact Match)} refers to the proportion of generated code that exactly matches the reference answers. This metric is highly stringent, as it does not consider partially correct cases and requires the generated code to be identical to the reference answer. The score can only be 0 or 1. The average number of exact matches out of 100 samples is calculated as the EM.

\textbf{ES (Edit Similarity)}, by contrast, measures the similarity between the generated code and the reference answer based on edit distance. This metric offers greater flexibility as it accounts for partially correct cases. However, it is less sensitive to minor differences.

For all generation tasks, we can use \textbf{generation time} to evaluate the effect of compression. We simply record the start and end times of generation. It is worth noting that some methods require preprocessing, retrieval, or compression of the data; the time spent on these stages is recorded as compression time separately and does not affect the generation time.

%rag 的EM表现比较好
\textit{Second}, in terms of code generation performance (EM), RAG-based methods often perform the best, with \rag outperforming \ragfunction. This could be because, compared to hard compression algorithms, RAG-based methods retain more context, and \rag, with its larger chunk size compared to \ragfunction, may preserve more potentially meaningful code information.

%硬压缩的时间表现比较
From the perspective of generation time, the hard compression methods—especially the \llmlingua family—perform quite well. This is because LLMLingua adopts a more fine-grained compression approach, as also evidenced by its high compression ratio. We also conduct an analysis of the relationship between the generation time and compression ratio, as shown in Figure \ref{fig:performance_compare}. It can be seen that these two metrics generally exhibit an inverse relationship; however, the trend does not perfectly match the theoretical expectation. This discrepancy may be attributed to factors such as cold start overhead and kv-cache behavior during LLM inference.


\textit{Finally}, we found that on the \ccevalpython dataset, the compression ratio of the RAG method can be less than 1. After analysis, we found that this is because the average sequence length of this dataset is relatively short, while the RAG window we set is relatively large, and we also used overlapping windows. As a result, the total number of input tokens after compression may actually increase. This suggests that we need to adjust the RAG window size based on the length of the input data for different datasets.

During the first part of our experiments, we observe that the compression ratio significantly influences compression performance (EM). This prompts us to investigate this issue further. Among existing methods, both \llmlingua and \longllmlingua allow for the specification of the compression ratio. We conduct experiments using both approaches with various setting ratios (i.e., 0.2, 0.4, 0.6, 0.8) across all datasets. 
The setting rate represents the ratio of the context after compression to the context before compression. Therefore, a lower setting rate indicates a higher compression ratio. In addition, considering the characteristics of the \nocompression and \nocontext methods, we use the \nocompression method to represent a setting rate of 1.0, while the \nocontext method corresponds to a setting rate of 0.0. Due to space limitations, we only present the results of the \llmlingua method on \lccpython, as shown in Table \ref{tab:experiment_ratio_lcc}. Since the setting compression rate does not fully reflect the actual compression rate, we also record the actual compression rate in the table.

Consistent with Table \ref{tab:experiment_ratio_lcc} highlights the experimental results with the highest EM, the shortest generation time, and the highest compression ratio in bold. It is important to note that the actual compression ratio is negatively correlated with the specified ratio. This occurs because, in the llmlingua method, the specified compression ratio represents the proportion of tokens after compression to those before compression, which is the opposite of our definition. 
We plot the performance of each compression method under different setting ratios as line charts, as shown in Figure \ref{fig:perfermance_ratio}. As shown in the Table \ref{tab:experiment_ratio_lcc} and Figure \ref{fig:perfermance_ratio}, we observe that as the specified ratio decreases, the actual compression ratio correspondingly increases, the generation time decreases, and the EM score also decreases. The trends of these three results are highly consistent with our expectations prior to the experiments.

However, we notice a phenomenon: for the Qwen2.5 model, when the specified compression ratio is set to 0.8, its EM score becomes lower than that obtained without any context. To investigate this further, we compare the context before and after \llmlingua compression, as shown in Figure \ref{fig:performance_compare}.

From the Figure~\ref{fig:performance_compare}, we observe that the compressed context becomes highly disordered, lacking any complete package names, function signatures, or variable information. For code generation tasks, such a disorganized context may have a negative effect and can even perform worse than providing no context at all. To investigate how models of different sizes respond to context compression, we conduct experiments on four models from the Qwen2.5-Coder family using various datasets. Table \ref{tab:experiment_size_lcc} and Figure \ref{fig:size} presents the results on the \lccpython dataset.
Analysis of Table \ref{tab:experiment_size_lcc} yields the following key findings.

First, regardless of the scenario, models with more parameters require longer generation times. This result aligns with our expectations, as many previous studies~\cite{Agarwal2023llm,Talamadupula2024aguide} have shown that when the model architecture and output length remain roughly constant, the inference time of a model tends to increase linearly with the number of parameters. 

Second, across all datasets, when no context compression is applied, the EM scores of Qwen2.5-Coder steadily increase as the parameter count grows from 0.5B to 1.5B, 3B, and 7B, demonstrating a pronounced parameter scaling law~\cite{kaplan2020scaling,wu2024towards}. This indicates that larger models can better understand and reason with a rich context, allowing their capabilities to be more fully realized.

Additionally, we observe that, under all compression methods, models with more parameters are more ``sensitive'' to compression, meaning that their accuracy suffers more. For example, using the highly compressive \llmlingua method, models of different sizes show little difference in EM scores on the \lccpython dataset, with smaller models even outperforming larger ones in some cases (e.g., the 1.5B model achieves a higher score than the 3B model). This suggests that the advantage of larger models is nearly eliminated under high compression ratios.

In contrast, moderate compression methods such as \rag and \ragfunction have a relatively smaller impact on large models. Compared to highly compressive methods like \llmlingua, these approaches maintain a reasonable compression rate without causing severe performance degradation in large models. This suggests that large models are able to effectively extract useful information when key information and structural elements are appropriately preserved in the compressed context.

Overall, these results indicate that large models can effectively capture useful information from long contexts, but high levels of compression cause significant loss of such information, which leads to a substantial decline in their performance. In contrast, small models have limited capacity to begin with and cannot capture long-range contextual information as effectively as large models; therefore, information loss due to compression has minimal impact on their performance~\cite{kaplan2020scaling}.

\section{Discussion}
% \section{Discussion \& IMPLICATIONS}
\vspace{-2pt}
In this section, we discuss the implications derived from the findings of our research questions and the threats to validity.

\subsection{Implications}
In the field of code, this trend is reflected in developers providing increasingly longer contexts to large models. Beyond function or class generation, some developers have already started exploring the use of LLMs for complete software generation tasks that can span several hours~\cite{manus,metagpt}.

However, excessively long contexts may introduce various issues, including, but not limited to, exceeding the LLM’s context window and causing truncation, substantially increasing generation time and negatively impacting user experience, and consuming too many tokens, thus leading to high costs. Therefore, it is necessary to compress long contexts.

We select seven representative compression methods and conduct experiments on eight models across three repository-level code completion datasets to analyze the performance of these compression algorithms on software engineering tasks. Based on our study, we obtain the following main findings:

First, models with similar numbers of parameters show little difference in performance on the same dataset and using the same method. This phenomenon aligns with the scaling law: if there are no substantial differences in data quality and training methodology, model capability is highly correlated with the number of parameters. Of course, different models may have their respective strengths and weaknesses with various methods, but this does not affect the overall conclusion.

Second, methods with lower compression factors result in longer generation times and correspondingly higher accuracy. This observation meets our expectations and is consistent with information theory. Shannon measures the average information content in a signal using entropy~\cite{shannon1948mathematical}. The theoretical limit of compression is entropy—it is impossible to recover the original data losslessly at a coding rate below the source entropy unless additional data structures are used, whereas all the methods we select are lossy. Thus, this phenomenon naturally occurs.

Additionally, we observe that models with fewer parameters are less affected by increasing the compression factor and, therefore, are relatively insensitive to compression. Upon examining the code generated by models of different sizes, we find that large models can more accurately capture useful information from long contexts in the absence of compression, while small models cannot. Since smaller models inherently struggle to extract effective information from lengthy contexts, compressing the context impacts them less.

Finally, we find that Java is less sensitive to compression than Python, which may be due to Java’s higher degree of redundancy. This finding is also supported by Shannon's information theory, which suggests that redundancy is critical for a system's resistance to noise and information loss. High redundancy enables the discarding of more non-essential information during compression without losing core content, resulting in less accuracy degradation~\cite{shannon1951prediction}. This is directly related to the code's structure and the redundancy inherent in the language itself.

\subsection{Threats to Validity}
\vspace{-2pt}
There are several potential threats to validity.
First, whether in experiments based on compression ratio or model size, we focus only on seven of the most widely used compression methods, which are mainly designed for natural language. As a result, our observations may not be generalizable to a broader range of methods, such as compression algorithms specifically designed for code. Nevertheless, the methods we selected are all influential and have high citation and star counts.

Additionally, the experimental parameters chosen for each method are not necessarily optimal, and therefore may not represent the best possible performance for all methods. To mitigate this threat, we conducted several rounds of pilot experiments and selected the current parameters based on those results.

Finally, the number of models we included is still limited and does not cover all code LLMs, such as earlier models like CodeBERT~\cite{feng2020codebert}. However, when selecting models, we took into account model size and release date. If we were to include a broader range of models, it would be unfair to models released several years ago with significantly fewer parameters than 7B. Therefore, we chose to evaluate only current mainstream models in our experiments.

\section{Related Works}
\vspace{-2pt}

The challenge of handling long context inputs in large language models has sparked extensive research in compression techniques, which can be broadly classified into three categories: RAG-based approaches, hard compression strategies, and soft compression methodologies.

\subsection{RAG-based Compression Approaches}
\vspace{-2pt}

RAG-based methods address context length limitations by selectively retrieving and incorporating only the most pertinent information into the input prompt~\cite{parvez2021retrievalaugmentedcodegeneration}. These approaches follow a three-stage pipeline: preprocessing, retrieval, and generation, with the primary innovation occurring in the retrieval mechanism. In repository-level code completion, RepoCoder~\cite{zhang2023repocoder} employs an iterative approach where generated completions inform subsequent retrieval rounds, creating a feedback loop that refines the selection process. Similarly, RLCoder~\cite{wang2024rlcoder} introduces a perplexity-based training strategy for the retriever, optimizing the selection of code blocks that maximize completion effectiveness. While these methods demonstrate strong performance in their respective domains, they require careful tuning of retrieval strategies and may struggle with novel or poorly-indexed content.

% \vspace{-2pt}
\subsection{Hard Compression Techniques}
Hard compression approaches directly modify the input by eliminating or reformulating less critical content. Early work in this area includes Selective Context~\cite{li-etal-2023-compressing}, LLMLingua~\cite{jiang2023llmlingua} and LongLLMLingua~\cite{jiang2023longllmlingua}. Selective Context~\cite{li-etal-2023-compressing}  focuses on contextually relevant segments. LLMLingua~\cite{jiang2023llmlingua} and LongLLMLingua~\cite{jiang2023longllmlingua} utilize learned knowledge and statistical importance measures to prune uninformative tokens and sentences. A notable advancement comes from LLMLingua-2~\cite{pan2024llmlingua}, which leverages data distillation from GPT-4 to train a specialized token classification model, achieving both efficiency and fidelity in compression. These methods offer the advantage of being model-agnostic and requiring no additional training, but may suffer from information loss in aggressive compression scenarios.

% \vspace{-2pt}
\subsection{Soft Compression Methodologies}
% \vspace{-2pt}
Soft compression techniques~\cite{chevalier2023adapting,mu2024learning,500xcompressor} take a fundamentally different approach by encoding input context into dense vector representations or prefix embeddings. These methods typically achieve superior compression ratios compared to hard compression alternatives, as they can capture semantic relationships more effectively. However, their practical applicability is limited by the requirement for training compression models that are tightly coupled with specific generative architectures, making them unsuitable for closed-source models such as GPT and Gemini.

% \vspace{-3pt}
\subsection{Code-Specific Compression}
% \vspace{-3pt}

While general-purpose compression methods struggle with code's unique properties, specialized approaches have emerged. DietCode~\cite{zhang2022diet} pioneered syntax-aware compression via AST analysis but suffers from architecture dependence. SlimCode~\cite{wang2024naturalbestmodelagnosticcode} improves upon this by preserving executability through structured AST pruning, while ShortenDoc~\cite{yang2024less} targets docstring compression. However, these methods remain limited in handling diverse languages and tasks efficiently, underscoring the need for more adaptable solutions that maintain accuracy under strict token budgets.

 % Recognizing that general-purpose compression methods often fail to capture the unique properties of code, researchers have developed specialized approaches for code compression. DietCode~\cite{zhang2022diet} represents an early effort, employing AST analysis to perform syntax-aware and semantic-preserving compression for code. However, its reliance on model-specific attention mechanisms limits its adaptability across different architectures. SlimCode~\cite{wang2024naturalbestmodelagnosticcode} advances this line of work by prioritizing code executability and compilability, using structured pruning based on AST analysis while maintaining program structure integrity. ShortenDoc~\cite{yang2024less} focuses on docstring compression, addressing a subset of the code compression challenge. Despite these advances, existing methods often lack the flexibility to handle diverse PLs and task requirements while maintaining both efficiency and accuracy. This  motivates the need for more robust solutions that can preserve task-relevant content under stringent token constraints.

\section{Conclusion}
% \vspace{4pt}

In this paper, we present the first exploratory study of long-context compression methods for source code, evaluating seven approaches across different large language models. We have conducted extensive experiments and our finding reveals a fundamental trade-off: while higher compression ratios improve efficiency, they consistently degrade code generation quality. This stems from current methods' inability to properly handle code-specific structures and semantics.
