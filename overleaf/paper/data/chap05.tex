\subsection{实验执行流程}
\label{subsec:run-pipeline}

整体执行流程见算法~\ref{alg:run-pipeline}。

\begin{algorithm}[H]
\caption{批量实验执行流程}
\label{alg:run-pipeline}
\KwIn{实例集合 $\mathcal{D}$（500 条），方法列表 $\mathcal{M}$，最大步数 $K=40$}
\KwOut{每方法的 resolve rate、avg tokens、CPR}

\BlankLine
\For{每种方法 $m \in \mathcal{M}$}{
  \For{每个实例 $\mathcal{T}_i = (r_i, s_i) \in \mathcal{D}$}{
    $\hat{s}_i \leftarrow \text{BuildContext}(s_i,\; r_i,\; m)$ \tcp*{按方法构建 task prompt}
    拉取 Docker 镜像 $\text{img}(r_i)$，挂载 \texttt{/testbed}\;
    将 \texttt{OPENAI\_BASE\_URL} 指向本地网关\;
    $\hat{p}_i \leftarrow \text{MiniSWEAgent}(\hat{s}_i,\; K)$ \tcp*{运行 agent，至多 K 步}
    $\text{resolved}_i \leftarrow \text{RunTests}(\hat{p}_i,\; \mathcal{T}_i)$ \tcp*{在 Docker 中执行测试}
    保存 \texttt{trajectory.json}，\texttt{result.json}，\texttt{patch.diff}\;
  }
  聚合统计：$\text{ResolveRate}(m)$，$\overline{C}(m)$，$\overline{K}(m)$，$\text{CPR}(m)$\;
}
\Return 所有方法的评估指标矩阵\;
\end{algorithm}

每个 instance 独立在 Docker 容器中运行，互不干扰。
所有方法共享相同的 mini-swe-agent 执行器和相同的测试评估逻辑，
区别仅在于 \texttt{BuildContext} 阶段注入的 task prompt 不同。

\subsection{评估指标}
\label{subsec:metrics}

遵循第~\ref{sec:cost-capability-tradeoff}~节的成本--能力框架，使用以下四个指标：

\begin{itemize}
  \item \textbf{Resolve Rate}：正确解决的 instance 比例（FAIL\_TO\_PASS 测试全部通过）
  \item \textbf{Avg Tokens}：所有 instance 的平均 \texttt{total\_tokens}（input + output）
  \item \textbf{Avg Steps}：平均执行步数
  \item \textbf{CPR}（Cost Per Resolved）：$\overline{C}_{\text{total}} \;/\; \text{ResolveRate}$，
        综合反映成本与能力的联合效率
\end{itemize}

\section{本地 Token 计量网关}
\label{sec:gateway}

\subsection{网关设计动机}
\label{subsec:gateway-motivation}

mini-swe-agent 通过 OpenAI-compatible API 调用 LLM。
为精确计量每一步的 token 消耗，而不依赖上游服务商的计费日志，
本文在 agent 与模型 API 之间插入一个\textbf{本地透明代理网关}（图~\ref{fig:gateway-arch}）。

\begin{algorithm}[H]
\caption{【占位图】此处应为本地 Token 计量网关架构图}
\label{fig:gateway-arch}
\tcc{图示内容：mini-swe-agent → 本地网关（FastAPI 透明代理）→ 上游 MiniMax API}
\tcc{网关拦截请求/响应，从 usage 字段提取 token 数并写入 JSONL 日志}
\tcc{同时将实验名、方法名、instance\_id 附加到每条日志记录}
\end{algorithm}

\subsection{网关实现}
\label{subsec:gateway-impl}

网关基于 \texttt{FastAPI} 实现，核心逻辑如下：

\begin{lstlisting}[style=promptstyle, caption={网关核心：拦截 token 用量并落盘（节选自 src/gateway/server.py）}, label={lst:gateway-core}]
@app.post("/v1/chat/completions")
async def chat_completions(request: Request) -> JSONResponse:
    body = await request.json()
    instance_id = request.headers.get("X-Instance-Id", "")
    strategy    = request.headers.get("X-Strategy-Name", "")

    # 转发至上游 API
    async with httpx.AsyncClient(timeout=180.0) as client:
        resp = await client.post(
            f"{upstream_url}/chat/completions",
            json=body,
            headers={"Authorization": f"Bearer {upstream_key}"},
        )

    data  = resp.json()
    usage = data.get("usage", {})

    # 记录 token 用量到 JSONL
    record = {
        "ts":           time.time(),
        "instance_id":  instance_id,
        "strategy":     strategy,
        "input_tokens":  usage.get("prompt_tokens",     0),
        "output_tokens": usage.get("completion_tokens", 0),
        "total_tokens":  usage.get("prompt_tokens", 0)
                         + usage.get("completion_tokens", 0),
    }
    with token_log_file.open("a") as f:
        f.write(json.dumps(record) + "\n")

    return JSONResponse(content=data)
\end{lstlisting}

启动时通过 YAML 配置文件指定上游 API 地址和模型名，
agent 侧只需将 \texttt{OPENAI\_BASE\_URL} 指向本地网关即可无感接入：

\begin{lstlisting}[style=promptstyle, caption={网关配置文件示例}, label={lst:gateway-config}]
upstream_url: "${MINIMAX_API_BASE_URL}"
upstream_key: "${MINIMAX_API_KEY}"
default_model: "MiniMaxAI/MiniMax-M2.5"
log_dir: "logs/gateway_minimax"
host: "127.0.0.1"
port: 8080
timeout: 180
\end{lstlisting}

每一步的 token 记录最终汇总写入 \texttt{trajectory.json} 的 \texttt{total\_usage} 字段，
作为后续分析的唯一 token 数据来源。

\section{方法选择动机与来源}
\label{sec:method-motivation}

\subsection{从 empirical 研究到系统优化实验}
\label{subsec:empirical-motivation}

本文所选取的六种优化方法（不含基线），并非随意拼凑，而是有明确研究依据的系统性选择。
其方法来源可以从两条线索加以理解。

\textbf{第一条线索：前期 empirical 研究的结论直接驱动了上下文压缩方法的选型。}
本课题组在前期工作~\cite{icse26longcontext}%
\footnote{该工作系本文作者参与的前期研究，已作为毕设前置基础，详见第~\ref{sec:prior-work}~节。}%
中，对七种长上下文代码压缩方法（包括滑动窗口 RAG、函数级 RAG、LLMLingua、
LongLLMLingua、LLMLingua-2 等）在代码补全任务上进行了系统性 empirical 评估，
涵盖四个代码大模型与三个仓库级数据集。
该研究揭示了两条核心结论，直接影响了本文的方法设计：

\begin{enumerate}
  \item \textbf{RAG 类方法在代码任务中效率与精度兼顾：}
        固定窗口 RAG 可将生成时间降低最多 67\%，同时精确匹配率（EM）几乎不下降；
        而 LLMLingua 等 token 级压缩方法虽可达到最高 10$\times$ 的压缩率，
        却在代码任务上造成 30--50\% 的 EM 下降。
  \item \textbf{代码结构使得通用压缩方法适配性不足：}
        代码中的缩进、类型注解、函数边界、调用关系等结构性 token，
        在语言学意义上看似"冗余"，却对语义理解具有不可替代的价值。
        通用 NLP 压缩方法往往无法甄别代码中的结构性语义，导致关键上下文丢失。
\end{enumerate}

基于上述结论，本文在仓库级代码修复场景中验证并扩展了这些发现：
选取文件级 BM25 检索（\texttt{rag\_topk}）作为代表性 RAG 方法，
在其基础上叠加函数级裁剪（\texttt{rag\_function}）以测试更激进的结构化压缩，
并引入 LLMLingua 语义重排序（\texttt{llmlingua\_original}）及其与 RAG 的组合（\texttt{hybrid\_llmlingua}），
以评估通用压缩方法在 agent 多轮修复场景中的真实表现。
换言之，上下文压缩类方法（方法二至四、七）的设计初衷，
是在 agent 场景下\textbf{重新验证 empirical 研究中的方法}，
并考察其在多轮动态上下文条件下是否仍然成立。

\textbf{第二条线索：成本分析框架的实验结论驱动了工作流压缩方法的原创设计。}
第~\ref{chap:framework}~章的阶段化成本分析表明，失败轨迹几乎总是耗尽 40 步预算，
是最主要的成本黑洞；代码阅读（understand）阶段占总 token 约 50\%，
而这一阶段的冗余本质上源于模型缺乏有效的搜索经验和任务规划能力。
这一发现指向了一个 RAG 方法无法直接解决的问题：
减少文件读入量可以压缩阅读成本，但若模型的决策质量低下、工作流混乱，
它仍会在已有候选文件中反复徘徊，产生无效的多步开销。

因此，本文针对上述瓶颈，设计并实现了两种\textbf{原创工作流压缩方法}：
结构化工作流脚手架注入（\texttt{skill\_abstraction}）
和静态可迁移经验记忆注入（\texttt{skill\_memory\_md}）。
这两种方法是本文的核心创新贡献，将在后续小节中详细说明其设计思路与实现。

\subsection{方法体系总览}
\label{subsec:method-overview}

综上，本文的七种方法构成两条正交优化路径：

\begin{itemize}
  \item \textbf{上下文压缩路径}（基于 empirical 研究方法移植与验证）：
        在 agent 运行前通过预检索精简输入上下文，直接压缩阅读阶段的 token 开销。
        包括：\texttt{rag\_topk}、\texttt{rag\_function}、\texttt{llmlingua\_original}、\texttt{hybrid\_llmlingua}。
  \item \textbf{工作流压缩路径}（本文原创方法设计）：
        通过向 agent 注入结构化决策指引或可迁移经验记忆，
        改善 agent 的工作流效率，减少失败轨迹中的无效 token 开销。
        包括：\texttt{skill\_abstraction}、\texttt{skill\_memory\_md}。
\end{itemize}

两条路径在机制上正交互补：前者优化"读什么"，后者优化"怎么做"。
两者的实验结果将在第~\ref{chap:results}~章联合分析，以揭示各自的成本—能力权衡特征。

\section{方法二：rag\_topk（文件级 RAG 检索）}
\label{sec:rag-topk}

\subsection{方法描述}

\texttt{rag\_topk} 在 agent 运行前，先根据 issue 文本从仓库中检索出少量候选文件，
并将这些文件的代码片段注入 task prompt。
其核心是：\textbf{用轻量规则检索替代 agent 的早期搜索阶段}，
减少 agent 在全仓库范围执行 \texttt{grep}/\texttt{find} 的 token 开销。

\subsection{实现流程}

完整流程见算法~\ref{alg:rag-topk}。

\begin{algorithm}[H]
\caption{rag\_topk：文件级 BM25 检索注入}
\label{alg:rag-topk}
\KwIn{问题描述 $s$，仓库快照 $r$（Docker \texttt{/testbed}），$k=4$，\texttt{snippet\_lines}=220}
\KwOut{增强后的 task prompt $\hat{s}$}

\BlankLine
\tcc{Step 1: 从 issue 中提取查询词}
$T \leftarrow \text{ExtractTerms}(s)$ \tcp*{提取标识符、类名、函数名等，去除停用词，取前 12 个}

\BlankLine
\tcc{Step 2: 在 Docker 中用 rg 检索候选文件}
$\text{cmd} \leftarrow$ \texttt{"rg -l -i -m 1 <pattern> . --glob '!*.json' | head -24"}\;
$\text{candidates} \leftarrow \text{DockerRun}(r,\; \text{cmd})$ \tcp*{最多 24 个候选文件路径}

\BlankLine
\tcc{Step 3: 读取候选文件前 snippet\_lines 行}
\For{每个候选文件 $f$}{
  $\text{content}[f] \leftarrow \text{DockerRun}(r,\; \texttt{"sed -n '1,220p' "} + f)$\;
}

\BlankLine
\tcc{Step 4: BM25 相关性打分，选 top-k 文件}
\For{每个候选文件 $f$}{
  $\text{score}[f] \leftarrow \text{BM25}(T,\; \text{content}[f]) + 2 \times \mathbb{1}[T \cap \text{path}(f) \neq \varnothing]$\;
}
$F_k \leftarrow \text{TopK}(\text{score},\; k=4)$\;

\BlankLine
\tcc{Step 5: 拼接候选文件片段到 prompt}
$\hat{s} \leftarrow s + \text{"Repository hints:"} + \bigcup_{f \in F_k} \text{"[Candidate File] "} + f + \text{content}[f]$\;
$\hat{s} \leftarrow \hat{s} + \text{"The additional context is a starting point..."}$\;
\Return $\hat{s}$\;
\end{algorithm}

\subsection{BM25 打分细节}

本文使用轻量自实现的 BM25（不依赖外部索引库），
查询词集合来自对 issue 文本的正则提取，并对文件路径中的关键词匹配给予额外加权：

\begin{lstlisting}[style=promptstyle, caption={TopKFileSelector BM25 打分（节选自 src/compression/context\_compressor.py）}, label={lst:bm25}]
def _bm25_score(query, files):
    query_terms = re.findall(r"\w+", query.lower())
    scores = {}
    for path, content in files.items():
        words = re.findall(r"\w+", content.lower())
        word_freq = {w: words.count(w) for w in set(words)}
        score = sum(word_freq.get(t, 0) for t in query_terms)
        # 路径匹配额外加权
        path_score = sum(2 for t in query_terms if t in path.lower())
        scores[path] = score + path_score
    return scores
\end{lstlisting}

\section{方法三：rag\_function（函数级裁剪检索）}
\label{sec:rag-function}

\subsection{方法描述}

\texttt{rag\_function} 在 \texttt{rag\_topk} 的基础上增加了\textbf{函数级裁剪}步骤：
对 top-k 候选文件，按 Python 顶层 \texttt{def}/\texttt{class} 边界切分代码块，
只保留与 issue 相关的函数和类，丢弃无关代码块，从而进一步减少注入的 token 数。

\subsection{函数级裁剪实现}

\begin{algorithm}[H]
\caption{FunctionLevelTrimmer：函数级代码裁剪}
\label{alg:func-trim}
\KwIn{文件内容 $C$，查询词集合 $T$}
\KwOut{裁剪后的文件内容 $C'$}

\BlankLine
$\text{blocks} \leftarrow \text{SplitByTopLevelDef}(C)$ \tcp*{按顶层 def/class 切分为块}
$T_{\text{exp}} \leftarrow \text{ExpandIdentifiers}(T)$ \tcp*{展开驼峰和下划线标识符}
$C' \leftarrow []$\;

\BlankLine
\For{每个代码块 $(t, \text{lines}) \in \text{blocks}$}{
  \uIf{$t = \texttt{import}$}{
    保留（import 语句始终保留）\;
  }
  \uElseIf{$t = \texttt{class\_sig}$}{
    保留类签名行\;
  }
  \uElseIf{$T_{\text{exp}} \cap \text{Terms}(\text{lines}) \neq \varnothing$}{
    保留（函数/类与查询词有交集）\;
  }
  \Else{
    丢弃（无关代码块）\;
  }
}
\Return $C'$\;
\end{algorithm}

标识符展开（\texttt{ExpandIdentifiers}）将驼峰命名和下划线命名拆分成子词，
例如 \texttt{separability\_matrix} 展开为 \{\texttt{separability}, \texttt{matrix}\}，
提高跨命名风格的匹配召回率。

\subsection{参数设置与风险}

\begin{table}[htbp]
  \centering
  \caption{rag\_function 参数配置}
  \label{tab:ragfunc-params}
  \begin{tabular}{lll}
    \toprule
    参数 & 值 & 说明 \\
    \midrule
    \texttt{top\_k\_files} & 4 & 同 rag\_topk \\
    \texttt{snippet\_lines} & 220 & 先截取前 220 行，再函数级裁剪 \\
    \texttt{enable\_function\_trim} & True & 启用函数级裁剪 \\
    \texttt{agent.max\_steps} & 40 & 最大执行步数 \\
    \bottomrule
  \end{tabular}
\end{table}

该方法的风险在于：若裁剪过激，可能丢失修复所需的辅助函数、类属性或导入关系，
导致 agent 在后续执行中仍需重新读取文件，抵消压缩效果。

\section{方法四：llmlingua\_original（语义文件重排序）}
\label{sec:llmlingua}

\subsection{方法描述}

\texttt{llmlingua\_original} 将 LLMLingua~\cite{jiang2023llmlingua} 的语义选择思想
适配到文件级检索场景。它不是对文件内容进行 token 级压缩，而是用
LLMLingua 的小型语言模型（\texttt{deepseek-coder-1.3b-base}）对候选文件进行
\textbf{语义相关性重排序}，以替代 BM25 的词频统计。

\subsection{LLMLingua 适配实现}

\begin{algorithm}[H]
\caption{llmlingua\_original：语义文件重排序}
\label{alg:llmlingua}
\KwIn{问题描述 $s$，候选文件集合 $F$（来自 rg 检索），$k=4$}
\KwOut{增强后的 task prompt $\hat{s}$}

\BlankLine
\tcc{Step 1: 用 rg 和关键词召回候选文件（同 rag\_topk）}
$F \leftarrow \text{CollectCandidates}(s, r)$\;

\BlankLine
\tcc{Step 2: 用 LLMLingua 对候选文件重排序}
$\text{compressor} \leftarrow \text{PromptCompressor}(\texttt{"deepseek-coder-1.3b"})$\;
\For{每个文件 $f \in F$}{
  将 $f$ 包装为 \texttt{SimpleNamespace(path=f, code\_content=content)}\;
}
$F_k \leftarrow \text{compressor.select\_cross\_files}(F,\; \text{question}=s,\; \text{topk}=4)$\;

\BlankLine
\tcc{Step 3: 同时启用函数级裁剪}
$F_k \leftarrow \{f: \text{FunctionTrim}(\text{content}[f], s) \;\text{for}\; f \in F_k\}$\;

\BlankLine
$\hat{s} \leftarrow s + \text{format}(F_k)$\;
\Return $\hat{s}$\;
\end{algorithm}

LLMLingua 的 \texttt{select\_cross\_files} 方法通过计算文件内容在问题条件下的
困惑度（perplexity）来评估文件的语义相关性，
相比 BM25 更能捕捉语义层面的关联，而非简单关键词共现。

\section{方法五：skill\_abstraction（工作流脚手架注入）}
\label{sec:skill-abstraction}

\subsection{设计背景与问题动机}
\label{subsec:skill-abs-motivation}

在基线实验中，agent 的执行轨迹分析揭示了一个普遍现象：
即使对于最终成功修复的 instance，agent 往往也要在搜索与阅读阶段消耗大量步骤，
反复在不同文件之间跳转，或在同一文件中多次确认已读过的内容，才能逐步收敛到正确修复位置。
对于失败的 instance，这一现象更加严重：agent 常常陷入一种"计划不清晰"的状态，
每一步都重新试探搜索方向，缺乏连贯的目标感，最终耗尽全部步数预算而无功而返。

这一现象的根本原因在于：大语言模型在面对开放式的仓库级修复任务时，
\textbf{缺乏内建的任务分解能力}。模型知道最终目标是"修复 issue"，
却没有一个清晰的心理模型引导其以最短路径到达目标。
在没有约束的情况下，模型倾向于逐步探索、反复确认，
而这种探索行为在 token 计价环境中代价高昂。

\texttt{skill\_abstraction} 的设计出发点正是对这一现象的直接回应：
通过向 task prompt 中注入一段\textbf{结构化工作流提示}（workflow scaffold），
为 agent 提供一个明确的六步求解框架，使其在整个修复过程中始终有清晰的阶段目标，
从而减少规划阶段的冗余推理，提升每一步行动的目的性。

\subsection{方法设计：结构化工作流脚手架}
\label{subsec:skill-abs-design}

工作流脚手架的核心是一段固定的自然语言指令，明确划定 agent 的六个执行阶段：

\begin{lstlisting}[style=promptstyle, caption={skill\_abstraction 注入的工作流提示（节选自 src/agent/task\_context.py）}, label={lst:skill-scaffold}]
Structured workflow:
1. Reproduce or understand the issue briefly.
2. Narrow to 1-3 candidate files.
3. Locate the exact function or block to edit.
4. Make the smallest patch that fixes the issue.
5. Verify with a focused check.
6. Submit the final patch only after verification.
\end{lstlisting}

这六步的设计并非随意排列，而是对成功修复轨迹的抽象提炼：

\begin{enumerate}
  \item \textbf{理解（Understand）}：先理解问题本质，避免过早跳入代码搜索。
        这一步鼓励 agent 在行动前进行语义归因，减少"搜索方向错误"导致的回溯代价。
  \item \textbf{定位候选文件（Narrow）}：将搜索范围收敛到 1--3 个文件，
        而非遍历整个仓库。这直接约束了搜索阶段的宽度，防止无限扩展。
  \item \textbf{精确定位（Locate）}：在候选文件内定位到函数或代码块级别，
        进一步压缩后续阅读的范围。
  \item \textbf{最小补丁（Minimal patch）}：明确要求生成最小化修改，
        防止 agent 生成过度重构的补丁，减少编辑—验证的迭代轮次。
  \item \textbf{验证（Verify）}：在提交前先做局部验证，而非依赖测试套件，
        降低验证成本。
  \item \textbf{提交（Submit）}：只有验证通过后才提交，避免无效提交引发的额外步骤。
\end{enumerate}

从成本分析角度看，这六步的编排方式体现了"漏斗式收敛"的思路：
每一步都将问题空间压缩到更小的范围，使后续步骤的搜索和阅读代价逐步降低。
与 baseline\_raw 相比，agent 不再需要在每一步自行决定"下一步该做什么"，
而是有一个清晰的路线图可以参照，从而将决策成本从"每步自主规划"转变为"沿路线图执行"。

\subsection{实现流程}

工作流脚手架注入在上下文检索之后执行：

\begin{algorithm}[H]
\caption{skill\_abstraction：工作流脚手架注入}
\label{alg:skill-abstraction}
\KwIn{问题描述 $s$，仓库快照 $r$}
\KwOut{增强后的 task prompt $\hat{s}$}

\BlankLine
\tcc{Step 1: 检索候选文件（同 rag\_topk）}
$F_k \leftarrow \text{RetrieveTopK}(s, r, k=4)$\;

\BlankLine
\tcc{Step 2: 构建 prompt：注入工作流提示 + 候选文件}
$\hat{s} \leftarrow s$\;
$\hat{s} \leftarrow \hat{s} + W_{\text{scaffold}}$ \tcp*{注入六步结构化工作流提示（固定字符串）}
$\hat{s} \leftarrow \hat{s} + \text{format}(F_k)$ \tcp*{追加候选文件片段}
$\hat{s} \leftarrow \hat{s} + \text{"The additional context is a starting point..."}$\;
\Return $\hat{s}$\;
\end{algorithm}

工作流提示被插入在 issue 文本之后、候选文件片段之前，
确保 agent 在看到代码上下文之前已经接收到工作流约束，
避免模型在读取到第一个候选文件后就立刻发起不受控的搜索行为。

\subsection{与上下文压缩方法的对比}

\texttt{skill\_abstraction} 与 RAG 类方法的本质区别在于：
RAG 方法优化的是"给 agent 看什么"（输入上下文的质量），
而工作流脚手架优化的是"agent 如何用它所看到的"（决策过程的质量）。
两者在机制上正交：
同一组候选文件，在有工作流约束时，agent 更可能按序完成定位—修改—验证流程；
在无约束时，即使文件已经正确，agent 也可能因为决策漂移而产生多余步骤。
因此，\texttt{skill\_abstraction} 可以与 RAG 方法叠加使用，
两者的优化目标不重叠。

\section{方法六：skill\_memory\_md（静态经验记忆注入）}
\label{sec:skill-memory}

\subsection{设计背景与核心思想}
\label{subsec:skill-mem-motivation}

\texttt{skill\_abstraction} 解决了 agent 缺乏工作流约束的问题，
但它注入的是"程序化的流程指令"，而非"来自实战经验的知识"。
在实际修复轨迹中，即便 agent 知道"应该在 1--3 个文件中定位问题"，
它在每类具体场景下究竟该如何搜索、应该避免哪些低效路径，仍是未知的。
这类"如何做"的隐性经验，无法通过固定流程指令传递，
而需要从历史成功案例中提炼和归纳。

\texttt{skill\_memory\_md} 正是基于这一思路设计的原创方法：
\textbf{将从已解决 instance 中归纳出的可迁移修复启发式经验，
以结构化文本（\texttt{skills.md}）的形式注入 agent 的 task prompt，
使其在面对新 instance 时能够复用这些"跨任务经验"}。

这一设计的灵感来自两个维度的观察。

\textbf{第一，从成功轨迹的阶段分析中发现规律性经验。}
在对已 resolved instance 的轨迹进行仔细分析后，可以归纳出若干可复用的经验模式：
例如，"对于描述参数验证行为的 issue，应优先从公共 API 入口点向内追溯"；
"当 issue 中明确出现测试函数名时，优先检查最近的测试模块以推断目标代码路径"；
"读取文件时避免整文件 dump，优先读取可疑行为所在函数，再视情况读一层调用方和被调用方"。
这些经验不是某个具体 instance 的答案，而是\textbf{可以跨越不同仓库、不同 issue 类型被复用的策略模式}。

\textbf{第二，从现有 agent 框架的记忆机制中获得启发。}
SWE-agent 和 AutoCodeRover 等框架已经证明，
向 agent 提供结构化的历史知识（工具文档、仓库摘要等）可以显著提升任务完成效率。
但这些框架中的知识通常是\textbf{instance 特定的}（针对当前仓库的结构信息），
而非\textbf{跨任务可迁移的}（从历史任务中提炼的策略经验）。
\texttt{skill\_memory\_md} 填补了这一空白：
它注入的是一套\textbf{与具体 instance 无关}的通用修复知识，
使 agent 在开始每个新任务时已具备一定的"修复经验积累"。

\subsection{skills.md 的构建方法论}
\label{subsec:skills-md-construction}

\texttt{skills.md} 的构建分为三个阶段：

\textbf{（1）轨迹采样与分析。}
从已完成实验的 resolved instance 中随机抽取代表性样本，
逐一阅读其 \texttt{trajectory.json} 中的工具调用序列，
重点关注：搜索阶段使用了哪些关键词、哪些搜索策略在最短步数内成功定位了目标文件、
阅读阶段选择了哪些函数粒度、编辑阶段采用了何种最小化修改策略、
验证阶段使用了什么命令。

\textbf{（2）跨 instance 归纳抽象。}
将具体操作提炼为与 instance 内容无关的策略描述，
去除仓库名、函数名、具体错误信息等 instance 特定元素，
保留可在新 instance 中复用的通用策略。
例如，将"在某仓库的代码文件中通过搜索精确错误类名找到入口"
抽象为"搜索 issue 中的精确错误消息名称或 API 名称，优先于搜索模糊关键词"。

\textbf{（3）分阶段组织写入 skills.md。}
将归纳出的启发式规则按照\texttt{SEARCH / READ / EDIT / VERIFY / SUBMIT} 五个阶段分节组织，
保证注入到 agent 的经验与其执行阶段相对应，便于模型在执行特定阶段时参照相关规则。

最终的 \texttt{skills.md} 包含五个模块，合计约 30 条启发式规则，
token 长度约 400--500（在 GPT-4o tokenizer 下），属于轻量级注入，
对总 token 预算的影响可忽略不计。

\subsection{skills.md 内容结构}

\texttt{skills.md} 的核心内容如下所示，涵盖跨任务可迁移的修复经验：

\begin{lstlisting}[style=promptstyle, caption={skills.md 内容节选（搜索与阅读启发式）}, label={lst:skills-md}]
# Transferable Skills For Mini-SWE-Agent
# Reusable heuristics distilled from previously resolved SWE-bench runs.

## Core Workflow
1. Identify 2-5 likely nouns, function names, classes, or modules from issue.
2. Use rg to narrow candidate files before any large file reads.
3. Once you have 1-3 plausible files, stop broad search; switch to local.
4. Make the smallest patch that changes behavior at the suspected fault site.
5. Run the narrowest verification first; submit once confirmed.

## Search Heuristics
- Search for exact error messages, API names, option names, and test names.
- If the issue describes argument validation, start near the public API
  entrypoint and trace one level inward.
- If tests are named in the issue metadata, inspect the nearest test module
  to infer target code paths.

## Read Heuristics
- Avoid whole-file dumps unless the file is short.
- Read the function containing the suspected behavior first, then one
  caller and one callee if needed.
- Do not re-read the same file unless checking a different symbol.

## Edit Heuristics
- Prefer minimal conditional changes, validation guards, or return-path
  fixes over broad refactors.
- Preserve existing style and error semantics unless the issue requires
  new behavior.

## Verification Heuristics
- After editing, run the narrowest targeted check that can falsify quickly.
- If targeted check passes, avoid broad test suites unless necessary.
\end{lstlisting}

这也是本文第~\ref{sec:hybrid}~节和第~\ref{chap:conclusion}~章展望中讨论的组合方向。

\section{方法七：hybrid\_llmlingua（组合压缩）}
\label{sec:hybrid}

\subsection{方法描述}

\texttt{hybrid\_llmlingua} 是本文设计的组合方法，将 LLMLingua 语义重排序、
函数级裁剪和 token budget 截断三个压缩步骤串联，
以最大化上下文压缩力度。它的目标是同时减少注入的文件数量和单文件的 token 长度。

\subsection{三阶段组合压缩流程}

\begin{algorithm}[H]
\caption{hybrid\_llmlingua：三阶段组合压缩}
\label{alg:hybrid}
\KwIn{问题描述 $s$，仓库快照 $r$，$k=4$，\texttt{token\_budget}=2500}
\KwOut{增强后的 task prompt $\hat{s}$}

\BlankLine
\tcc{阶段一：rg 召回候选文件（同 rag\_topk）}
$F \leftarrow \text{CollectCandidates}(s, r)$ \tcp*{最多 24 个候选}

\BlankLine
\tcc{阶段二：LLMLingua 语义重排序，选 top-k}
$F_k \leftarrow \text{LLMLinguaRank}(F,\; s,\; k=4)$\;

\BlankLine
\tcc{阶段三：函数级裁剪}
\For{每个文件 $f \in F_k$}{
  $\text{content}[f] \leftarrow \text{FunctionTrim}(\text{content}[f],\; s)$\;
}

\BlankLine
\tcc{阶段四：token budget 截断}
$F_k \leftarrow \text{TokenBudgetTrim}(F_k,\; \text{budget}=2500)$\;
\tcp*{按文件长度升序依次填入，超出 budget 则截断或跳过}

\BlankLine
$\hat{s} \leftarrow s + \text{format}(F_k)$\;
\Return $\hat{s}$\;
\end{algorithm}

\subsection{token budget 截断逻辑}

\texttt{TokenBudgetTrimmer} 按文件长度（token 数）从短到长依次填入：

\begin{lstlisting}[style=promptstyle, caption={TokenBudgetTrimmer 截断逻辑（节选自 src/compression/context\_compressor.py）}, label={lst:budget-trimmer}]
def trim(self, files, query):
    result, used = {}, 0
    sorted_files = sorted(files.items(),
                          key=lambda kv: count_tokens(kv[1]))
    for path, content in sorted_files:
        tokens    = count_tokens(content)
        remaining = self.token_budget - used
        if tokens <= remaining:
            result[path] = content
            used += tokens
        elif remaining > 200:
            enc       = tiktoken.get_encoding("cl100k_base")
            truncated = enc.decode(enc.encode(content)[:remaining])
            result[path] = truncated + "\n# [TRUNCATED BY TOKEN BUDGET]"
            used += remaining
        if used >= self.token_budget:
            break
    return result
\end{lstlisting}

\subsection{参数设置}

\begin{table}[htbp]
  \centering

\subsection{风险分析}

多重压缩步骤叠加后，存在\textbf{过度压缩}的风险：
若 LLMLingua 重排序遗漏了关键文件，或函数级裁剪丢弃了必要的辅助函数，
再经过 token budget 截断，最终注入的上下文可能已不包含修复所需信息。
此时 agent 仍需自行补充搜索，反而增加了总 token 消耗。
因此该方法需要通过完整实验验证，确认组合压缩是否真正优于单一压缩。

\section{本章小结}
\label{sec:chap05-summary}

本章介绍了完整的实验平台与七种成本优化方法的实现细节，主要内容包括：

\begin{enumerate}
  \item \textbf{实验平台}：基于云服务器、官方 SWE-bench Docker 镜像和
        mini-swe-agent 构建统一执行环境，500 条 SWE-bench Verified instance
        在七种方法下独立运行，保证可比性。

  \item \textbf{token 计量网关}：设计本地 FastAPI 透明代理，
        精确记录每步 \texttt{input\_tokens} 和 \texttt{output\_tokens}，
        为成本分析提供统一数据来源。

  \item \textbf{上下文压缩方法}（\texttt{rag\_topk}、\texttt{rag\_function}、
        \texttt{llmlingua\_original}）：通过关键词检索 + BM25 打分 / LLMLingua
        语义重排序 + 函数级裁剪，在 agent 运行前注入精简的候选文件上下文，
        目标是减少搜索和阅读阶段的 token 消耗。

  \item \textbf{工作流压缩方法}（\texttt{skill\_abstraction}、\texttt{skill\_memory\_md}）：
        通过注入结构化工作流提示或静态跨任务经验记忆，引导 agent 更快进入
        有效编辑阶段，减少规划空转。

  \item \textbf{组合压缩方法}（\texttt{hybrid\_llmlingua}）：
        串联语义重排序、函数裁剪和 token budget 截断，追求最大压缩力度，
        同时承担过度压缩的风险。
\end{enumerate}

各方法的实际效果将在第~\ref{chap:results}~章通过完整实验数据进行对比分析。
