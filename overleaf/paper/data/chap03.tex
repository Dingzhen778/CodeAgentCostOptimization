\chapter{相关工作与研究基础}

本章围绕本文的研究问题，对自动程序修复、软件工程智能体、仓库级代码任务、长上下文代码理解与压缩、成本优化与效率分析等方向的代表性工作进行系统梳理。与"实验综述"或"方法说明"不同，相关工作的核心任务并不是提前给出本文实验结论，而是回答三个更基础的问题：第一，已有研究如何逐步把程序修复从局部代码变换推进到真实仓库级问题求解；第二，现有系统在上下文获取、工具调用和多轮决策方面已经发展到了什么程度；第三，面对越来越真实的 GitHub issue resolution 任务，为什么"成功率之外的推理成本"正在成为一个必须单独研究的问题。基于这些问题，本章将进一步明确本文与现有研究的关系，并说明前期工作如何自然融入当前毕设主线。

\section{基于大语言模型的自动程序修复}

\subsection{传统自动程序修复方法的研究脉络}

自动程序修复（Automated Program Repair，APR）并非大语言模型时代才出现的新问题。早期研究已经围绕"如何自动构造候选补丁"和"如何验证候选补丁是否有效"形成了较为完整的技术谱系。总体上，传统 APR 大多遵循 generate-and-validate 的基本范式：先在某个补丁空间中生成候选补丁，再利用测试套件、静态约束或语义条件筛选补丁。围绕这一基本范式，研究者逐步形成了搜索式、模板式和语义式三条主要路线。

搜索式 APR 以 GenProg~\cite{le2011genprog} 为代表，它将程序修复建模为程序变异与搜索问题，通过对抽象语法树施加删除、插入、替换等操作生成候选补丁，再用测试套件筛除错误结果。其核心算法流程如算法~\ref{alg:genprog} 所示。

\begin{algorithm}[H]
\caption{GenProg: 基于遗传编程的自动程序修复}
\label{alg:genprog}
\KwIn{缺陷程序 $P$，测试套件 $T = \{t_1, t_2, \ldots, t_n\}$，最大代数 $G$}
\KwOut{修复补丁 $patch$ 或失败标志}
初始化种群 $\text{Pop} \leftarrow \{P\}$\;
\For{$g = 1, 2, \ldots, G$}{
  \For{每个个体 $p \in \text{Pop}$}{
    $f(p) \leftarrow |\{t \in T : p \text{ 通过 } t\}|$\;
  }
  \If{$\exists\, p^* \in \text{Pop}$ 使得 $f(p^*) = |T|$}{
    \Return $\text{diff}(P,\, p^*)$\;
  }
  通过锦标赛选择适应度高的个体\;
  对 AST 节点施加随机变异（删除 / 插入 / 替换）\;
  $\text{Pop} \leftarrow \text{crossover}(\text{Pop}) \cup \text{mutate}(\text{Pop})$\;
}
\Return \textbf{失败}\;
\end{algorithm}

GenProg 的重要意义不在于它给出了唯一正确的搜索算法，而在于它证明了"在真实程序上自动搜索补丁"本身是可行的。随后，大量工作围绕搜索空间构造、缺陷定位和候选排序展开，逐步推动了基于测试驱动修复的研究社区形成。尽管如此，搜索式方法始终面临两个难题：一是补丁空间随程序规模迅速膨胀，二是"通过测试"并不必然意味着语义正确，因此容易出现过拟合补丁。

模板式 APR 则试图利用已有修复经验直接缩小搜索空间。TBar~\cite{liu2019tbar} 是此方向的代表性工作，它统一实现并评估了大量 fix patterns，展示了模板式方法在常见缺陷类型上的实用价值。与搜索式方法相比，模板式方法的优势在于生成空间更受控、补丁更易解释、实现更接近开发者真实修改习惯。然而，这类方法高度依赖模板覆盖范围，面对跨文件依赖、复杂控制流或新型错误模式时，往往难以保持稳定效果。从研究脉络上看，模板式方法的重要贡献是把"补丁先验"明确编码进系统，为后续学习式和大模型式修复提供了对照基线。

语义式 APR 进一步尝试从程序语义层面对修复进行约束。SemFix~\cite{nguyen2013semfix} 通过符号执行、约束求解和程序合成生成满足语义条件的修复表达式，是该方向中最具代表性的工作之一。这一路线的优势在于，它不再只依赖外部测试通过作为唯一信号，而是显式利用程序路径和语义约束提高补丁正确性。不过，语义式方法的扩展性长期受到限制：一旦程序规模增加、状态空间变复杂，符号执行和约束求解成本就会快速上升，难以直接覆盖开放世界的仓库级维护场景。

总体而言，传统 APR 为后续研究奠定了三个重要基础。首先，它明确了补丁生成与补丁验证的基本任务结构；其次，它揭示了候选空间设计、验证成本和补丁过拟合是程序修复中长期存在的共性问题；最后，它提供了大量评测基准和经典问题设定，使后来的学习式 APR 与 LLM-based APR 能够在统一语境下比较。从本文角度看，传统 APR 的价值更多体现在问题演化脉络和方法学基础，而不是直接提供针对大模型推理成本的解决方案。

\subsection{从学习式 APR 到大语言模型驱动修复}

在传统 APR 之后，程序修复研究开始逐渐引入数据驱动与学习式方法。Defects4J~\cite{chen2021evaluating} 等基准的广泛使用，为学习式 APR 提供了相对稳定的训练和评测环境。神经机器翻译、序列到序列模型和预训练代码模型的引入，使"修复"逐步从模板实例化问题，转向基于数据学习补丁变换规律的问题。CodeBERT~\cite{feng2020codebert}、CodeT5~\cite{wang2021codet5} 等代码预训练模型虽然并非专门为 APR 设计，但它们为错误定位、补丁生成和代码理解提供了更强的表示能力，也推动了程序修复从"规则驱动"向"表征驱动"的转变。

真正使 APR 研究范式发生明显变化的是大语言模型的出现。Xia 等人在 ICSE 2023 的研究《Automated Program Repair in the Era of Large Pre-trained Language Models》~\cite{xia2023automated} 系统评估了多种大模型在 APR 场景中的表现，比较了整段生成、单行修复、infilling 等不同设置，表明大模型已经可以在不少场景中生成有竞争力的修复补丁。这一结果的重要意义，不只是"模型更强了"，而是修复系统的研究重点开始从"如何人工设计补丁空间"转向"如何组织上下文，让模型更稳定地生成正确补丁"。

在这一阶段，单轮补丁生成是最直接也最常见的做法。系统通常把 issue 描述、失败测试、待修复代码片段以及有限范围的上下文拼接为 prompt，然后要求模型直接输出修复后的代码，如图~\ref{lst:apr-prompt} 所示。

\begin{lstlisting}[style=promptstyle, caption={典型 LLM-based APR 单轮 prompt 示例}, label={lst:apr-prompt}]
### System
You are an expert software engineer. Fix the bug in the function below.

### Issue description
TypeError: unsupported operand type(s) for +: 'int' and 'str'

### Failing test
def test_add():
    assert add(1, "2") == 3  # expects int conversion

### Buggy function
def add(a, b):
    return a + b  # bug: no type coercion

### Fixed function:
def add(a, b):
    return a + int(b)
\end{lstlisting}

该流程实现简单，便于与传统 APR 进行对照，因此在早期工作中被广泛采用。然而，随着任务从局部函数修复扩展到复杂仓库级问题，单轮生成的局限很快显现：模型既可能缺乏足够的仓库级上下文，也难以在没有外部反馈的情况下自我校正错误。尤其是涉及跨文件依赖、历史实现约定和配置语义时，一次性补丁生成往往只会产生"看似合理"的代码，而无法保证其在真实执行环境下通过验证。

为了提升大模型修复的稳定性，研究者开始尝试更细粒度的输入组织方式。Infilling 允许模型在保留前后文锚点的情况下补全中间缺失代码，比整段重写更适用于局部修改场景。结构化输入组织则尝试把缺陷位置、相关测试、上下文代码和任务指令显式分块，以减少模型对无关噪声的敏感性。与此同时，补丁排序、候选重采样和自我验证等机制也被引入 APR 流程，试图缓解单次生成的偶然性问题。虽然这些工作已经显著提升了补丁生成的可用性，但它们仍大多建立在"给定固定上下文"的前提之上，对本文关注的动态过程成本问题涉及有限。

\subsection{对话式修复与交互式程序修复}

随着研究者逐渐认识到"修复不是一次性生成任务"，交互式和对话式程序修复开始成为重要分支。ChatRepair~\cite{xia2023keep} 是这一方向的典型代表。该工作将编译失败、测试失败等反馈重新送回模型，让模型在多轮对话中不断修正前一轮补丁，从而显著提高修复成功率。其算法流程如算法~\ref{alg:chatrepair} 所示。

\begin{algorithm}[H]
\caption{ChatRepair: 多轮对话式程序修复}
\label{alg:chatrepair}
\KwIn{缺陷程序 $P$，测试套件 $T$，最大轮次 $K$}
\KwOut{修复补丁或失败标志}
$h \leftarrow \bigl[(\texttt{system:}\ \text{APR expert}),\;(\texttt{user:}\ \text{issue} + \text{buggy code})\bigr]$\;
\For{$k = 1, 2, \ldots, K$}{
  $patch_k \leftarrow \text{LLM}(h)$\tcp*{模型生成补丁}
  $result \leftarrow \text{execute\_tests}(P \oplus patch_k,\; T)$\;
  \If{$result = \textsc{AllPass}$}{
    \Return $patch_k$\;
  }
  $feedback \leftarrow \text{format\_feedback}(result)$\tcp*{编译错误 / 测试失败详情}
  $h \leftarrow h + \bigl[(\texttt{asst:}\ patch_k),\;(\texttt{user:}\ feedback)\bigr]$\;
}
\Return \textbf{失败}\;
\end{algorithm}

与传统 generate-and-validate 相比，对话式修复并没有放弃"生成 + 验证"的基本结构，而是把验证结果直接纳入下一轮生成过程，使补丁优化从单次试探演化为多轮闭环。这类工作说明了一个关键事实：在大模型程序修复中，外部环境反馈是模型生成能力的重要补充。测试失败不仅提供了"补丁没成功"的二元信号，还能作为线索帮助模型修正理解偏差；编译错误则往往能暴露出语法、接口调用或命名上的明显问题。多轮修复机制因此更接近真实开发过程，也解释了为什么仓库级修复越来越倾向于 agent 式实现。

但交互式修复也带来了新的工程代价。随着对话轮次增加，历史 observation、错误日志、旧补丁和中间推理会不断进入上下文，使后续轮次的输入长度持续增长。换言之，提升修复能力的机制本身同时创造了新的成本来源。ChatRepair 等工作已经开始报告单位 bug 的 token 或美元成本，这表明 APR 社区已逐渐意识到"成功率提升"并不是免费的。从本文角度看，这一转变非常关键，因为它把"修复效果"和"修复代价"首次放到了同一评价框架中。

\subsection{面向真实场景的 APR 研究趋势}

综合上述工作可以看到，APR 正在经历两条并行演化。第一条是方法能力的提升，即从局部搜索和模板修复，逐步过渡到基于大模型的上下文驱动生成，再进一步发展到引入测试反馈、工具执行和多轮决策的交互式修复。第二条是任务环境的外延，即从结构较受控的函数级、文件级缺陷，逐渐扩展到包含仓库上下文、真实 issue 描述和工程依赖的开放场景。

这种演化趋势意味着，程序修复的核心难点正在变化。传统 APR 的主要挑战是如何在有限空间中找到有效补丁，而大模型时代的主要挑战越来越多地体现在：如何准确理解仓库、如何控制上下文噪声、如何避免多轮过程中的信息膨胀，以及如何在有限预算下保持足够的修复能力。因此，本文关注的"代码修复智能体高推理成本问题"并不是脱离 APR 主线的附加问题，而是随着 APR 场景日益真实化而自然浮现出的新研究重点。


\section{代码智能体与软件工程智能体}

\subsection{从单轮代码生成到工具调用式智能体}

代码生成研究最初多把模型视为"给定输入，生成输出"的单轮系统，但软件工程任务的复杂性很快暴露出这种范式的局限。真实软件开发和维护过程并不只是写一段代码，而是需要阅读代码库、定位问题、搜索实现、执行测试、查看日志、编辑文件并反复验证。随着研究场景从代码补全扩展到问题定位和仓库级修复，模型逐渐被置于一个需要与外部工具持续交互的环境中，代码智能体由此成为更自然的系统形态。

在方法论层面，ReAct~\cite{yao2023react} 和 Toolformer~\cite{schick2023toolformer} 等工作虽然并非专门面向软件工程，但它们为后续代码智能体提供了重要启发：大模型不应只负责输出最终答案，还应显式组织推理过程、调用外部工具并根据工具返回结果动态调整下一步行动。在软件工程场景中，这一思想尤其重要，因为仓库阅读、文件搜索和测试执行本身就是问题求解的一部分，而不是外部于推理过程的辅助步骤。

\subsection{软件工程智能体的代表性系统}

SWE-agent~\cite{yang2024sweagent} 是当前软件工程智能体研究中最具代表性的系统之一。它提出 agent-computer interface（ACI）的概念，强调代码编辑、文件浏览、命令执行和上下文管理接口的设计，会直接影响模型完成真实软件工程任务的能力。SWE-agent 的核心执行循环如算法~\ref{alg:sweagent} 所示。

\begin{algorithm}[H]
\caption{SWE-agent: ACI 执行循环}
\label{alg:sweagent}
\KwIn{GitHub issue $I$，仓库环境 $\mathcal{E}$，LLM $\mathcal{M}$，最大步数 $N$}
\KwOut{补丁文件或失败标志}
$obs_0 \leftarrow \textsc{InitEnv}(\mathcal{E})$\tcp*{初始化环境，获取仓库目录概览}
$ctx \leftarrow \bigl[(\texttt{system:}\ \text{ACI instructions}),\;(\texttt{user:}\ I + obs_0)\bigr]$\;
\For{$n = 1, 2, \ldots, N$}{
  $action_n \leftarrow \mathcal{M}(ctx)$\tcp*{模型预测下一步 ACI 命令}
  \If{$action_n = \texttt{submit}$}{
    \Return $\textsc{Diff}(\text{原始仓库},\;\text{当前仓库状态})$\;
  }
  $obs_n \leftarrow \mathcal{E}.\textsc{Execute}(action_n)$\tcp*{执行命令，获取工具观测}
  $ctx \leftarrow ctx + \bigl[(\texttt{asst:}\ action_n),\;(\texttt{user:}\ obs_n)\bigr]$\;
}
\Return \textbf{失败（超出最大步数）}\;
\end{algorithm}

SWE-agent 的 ACI 工具集包括 \texttt{find\_file}、\texttt{search\_dir}、\texttt{open}、\texttt{edit}、\texttt{run\_tests} 等命令，并通过严格的上下文窗口截断策略控制输入长度。其完整工具接口设计（含参数说明和示例输出）请参见原论文 Yang et al.~\cite{yang2024sweagent} Figure 1 及附录。

AutoCodeRover~\cite{zhang2024autocoderover} 进一步强化了"问题定位 + 代码修改"的拆分思路。该系统强调在真正改动代码之前，先通过搜索、调试和相关文件定位缩小问题空间，从而让后续修改更有针对性。其层次化搜索流程如算法~\ref{alg:autocoderover} 所示。

\begin{algorithm}[H]
\caption{AutoCodeRover: 层次化搜索定位 + 补丁生成}
\label{alg:autocoderover}
\KwIn{GitHub issue $I$，仓库 $\mathcal{R}$，LLM $\mathcal{M}$，最大重试次数 $R$}
\KwOut{修复补丁或失败}
\For{$r = 1, \ldots, R$}{
  \tcc{第一层：文件级搜索}
  $F \leftarrow \mathcal{M}.\textsc{SearchFiles}(I,\;\mathcal{R}.\text{structure})$\;
  \tcc{第二层：符号级搜索（类 / 方法 / 属性）}
  $S \leftarrow \mathcal{M}.\textsc{SearchSymbols}(I,\; F)$\;
  \tcc{第三层：上下文提取与组织}
  $\text{Context} \leftarrow \textsc{ExtractContext}(S,\;\mathcal{R})$\;
  \tcc{补丁生成}
  $patch \leftarrow \mathcal{M}(I,\;\text{Context})$\;
  \If{$\textsc{Validate}(patch,\;\mathcal{R})$}{
    \Return $patch$\;
  }
  \tcc{将错误反馈加入下一轮搜索提示}
}
\Return \textbf{失败}\;
\end{algorithm}

Agentless~\cite{xia2024agentless} 则从另一个方向挑战了复杂 agent 脚手架的必要性。它将 issue resolution 简化为"定位—修复—验证"的较固定三阶段流程，如算法~\ref{alg:agentless} 所示。

\begin{algorithm}[H]
\caption{Agentless: 结构化三阶段 Issue Resolution}
\label{alg:agentless}
\KwIn{GitHub issue $I$，仓库 $\mathcal{R}$，LLM $\mathcal{M}$，采样次数 $S$}
\KwOut{修复补丁}
\tcc{阶段一：缺陷定位}
$\text{FileList} \leftarrow \mathcal{M}(I,\;\mathcal{R}.\text{structure})$\tcp*{预测相关文件}
$\text{LocList} \leftarrow \mathcal{M}(I,\;\text{FileList content})$\tcp*{预测待修改行 / 函数}
\tcc{阶段二：多次采样生成补丁}
$\text{Patches} \leftarrow \emptyset$\;
\For{$s = 1, \ldots, S$}{
  $p_s \leftarrow \mathcal{M}(I,\;\text{LocList context})$\;
  $\text{Patches} \leftarrow \text{Patches} \cup \{p_s\}$\;
}
\tcc{阶段三：补丁排序与选择（多数投票 / 格式过滤）}
$p^* \leftarrow \textsc{RankAndSelect}(\text{Patches})$\;
\Return $p^*$\;
\end{algorithm}

Agentless 的研究结论与本文高度相关：过于自由的智能体规划并不一定优于结构化流程，有时反而会带来额外的系统开销和不可控行为，这说明"智能体越复杂越强"并不是必然规律。

在更复杂的系统设计上，多智能体框架如 MAGIS~\cite{tao2024magis} 和 CodeR~\cite{chen2024coder} 进一步探索了角色分工、任务图和协同决策在软件工程问题中的作用。这类系统通常让不同 agent 分别承担仓库理解、开发、测试、质量保证等职责，试图借助角色分工提升任务完成率。其研究价值在于，它们揭示了软件工程任务中的确存在天然可分解结构；但它们同样会引入更高的协调成本和更长的交互轨迹。对本文而言，多智能体工作最重要的意义不是作为直接实现方案，而是作为说明 agent overhead 可能随系统复杂度上升而显著放大的现实例证。

\subsection{软件工程智能体研究的核心问题}

现有软件工程智能体研究至少围绕四个关键问题展开。第一，如何为模型提供合适的工具接口，使其能有效浏览仓库、执行命令和编辑文件。第二，如何把真实软件任务拆成更适合模型处理的子阶段，例如问题定位、相关文件检索、补丁生成和测试验证。第三，如何管理多轮执行过程中不断增长的上下文，避免早期 observation 变成后续负担。第四，如何评估一个智能体系统，不仅比较其是否成功，还比较其实现成功所消耗的资源。

其中，前三个问题大多已经在不同系统中得到一定程度的讨论；第四个问题则仍处于相对初步的阶段。当前主流论文往往优先报告 resolve rate、pass rate 或 benchmark 排名，而较少深入拆解轨迹长度、输入 token、失败重试代价和不同工具调用对总成本的贡献。正因如此，代码智能体研究虽然已经非常强调"真实环境执行"，但对"真实环境中的成本结构"仍缺乏统一而系统的刻画，这正为本文留下了明确的研究空间。


\section{仓库级代码任务与 GitHub Issue Resolution}

\subsection{从函数级代码生成到仓库级任务}

代码任务评测最初主要集中在函数级或短程序级别。HumanEval~\cite{chen2021evaluating} 和 MBPP 等基准的重要贡献在于，它们以可执行测试为评价标准，推动了代码生成研究从自然语言质量比较转向"程序能否真正运行"的功能性评价。然而，这类任务通常只涉及单个函数或短小程序，依赖的上下文有限，难以反映真实软件工程中跨文件依赖、历史实现约定和项目结构理解等需求。

这一阶段的任务虽然尚未直接进入 issue resolution，但已经奠定了两项关键认识。其一，真实软件任务需要跨文件依赖信息，而不是仅靠局部片段。其二，上下文越大并不意味着问题越容易解决，因为相关信息往往稀疏分布在大量无关代码中。也正是在这种背景下，长上下文代码理解与上下文选择问题逐渐成为独立研究主题。

\subsection{GitHub Issue Resolution 的提出与意义}

SWE-bench~\cite{jimenez2024swebench} 将任务进一步推进到真实 issue resolution 场景。与函数补全或局部修复不同，SWE-bench 给定的是完整仓库、真实 GitHub issue 描述以及用于验证修复结果的测试环境，要求系统生成补丁并通过测试。图~\ref{lst:swebench-instance} 展示了一个典型的 SWE-bench 数据集实例格式。

该基准的重要性在于，它首次把"真实软件维护"大规模地引入大模型评测体系，使程序修复、代码理解、仓库浏览和环境交互等能力在同一任务中发生耦合。从研究视角看，issue resolution 相较于传统 APR 和仓库级补全有两点根本差异。第一，它的输入不再只是代码和测试，还包括自然语言 issue 描述，其中包含问题背景、现象描述、期望行为乃至模糊表达。第二，它要求系统在开放仓库环境中自行决定应读哪些文件、该做哪些修改以及如何验证结果，因此更接近真实开发者的维护流程。这意味着 issue resolution 天然适合作为代码智能体的评测场景，也天然更容易暴露上下文管理和推理成本问题。

\subsection{SWE-bench 生态及其衍生基准}

随着 SWE-bench 被广泛采用，研究社区很快意识到其评测成本和数据质量问题。一方面，完整 issue resolution 评测依赖仓库环境构建、测试执行和多轮尝试，成本远高于函数级任务；另一方面，原始数据中也可能存在任务描述不清晰、测试补丁不稳定或问题已难以复现等情况。因此，Lite 和 Verified 等衍生子集相继出现。SWE-bench Lite 通过更小的子集降低实验门槛，使系统迭代和消融验证更加可行；SWE-bench Verified 则通过人工筛选与验证提升 benchmark 的可解性和比较可靠性。

这些衍生基准的出现，对于本文的研究问题具有直接意义。它们表明，研究社区已经从 benchmark 设计层面承认"真实软件工程任务的评测是昂贵的"，因此不仅要关心方法本身是否有效，也要关心评测流程和任务选择是否可承受。换言之，成本问题不仅存在于模型推理阶段，也已经体现在 benchmark 建设和实验组织层面。

\subsection{Issue Resolution 场景对本文研究的启示}

综合来看，从 HumanEval 到 RepoBench，再到 SWE-bench 及其多语言、多模态扩展，代码任务的发展路径可以概括为"上下文越来越长，任务越来越开放，环境越来越真实"。这一演化直接催生了本文的研究动机。对于函数级基准而言，成本更多表现为单次生成成本；而对 issue resolution 而言，成本是一个贯穿整个执行过程的累计量，涉及搜索、阅读、编辑、测试、重试以及历史 observation 管理。因而，若仍沿用早期代码生成任务中的评价方式，仅比较最终成功率，就很难准确反映真实系统的工程价值。


\section{长上下文代码理解与上下文压缩}

\subsection{长上下文代码任务的形成}

仓库级任务的出现使"长上下文代码理解"从附属问题转变为核心研究方向。早期研究更多关心代码生成能力本身，而仓库级任务要求模型从大规模无关代码中识别真正相关的实现、接口和依赖关系，因而更接近信息检索与程序理解的结合问题。RepoQA 将这一点表达得尤为明确：它不再只测模型能否从长上下文中复制某段代码，而是要求模型完成对仓库结构有意义的"找函数"式理解任务，从而衡量模型在长仓库上下文中的搜索与理解能力。

随着模型上下文窗口不断扩大，围绕长代码理解的 benchmark 也逐渐丰富。LONGCODEU、LongCodeBench、LoCoBench 和 Long Code Arena 等工作进一步把评测推向更长输入尺度和更多软件开发任务维度，覆盖修复、生成、检索、总结等多种场景。它们共同揭示出一个越来越清晰的事实：模型拥有更长的上下文窗口，并不意味着它自动具备稳定的长代码理解能力。尤其在代码长度超过若干阈值后，性能往往出现明显退化，这说明长上下文问题不仅是"能不能塞进去"，更是"能不能有效利用"的问题。

\subsection{通用 Prompt 压缩方法}

在长上下文压力增加后，通用 prompt compression 迅速成为热门研究方向。LLMLingua~\cite{jiang2023llmlingua}、LongLLMLingua~\cite{jiang2023longllmlingua} 和 LLMLingua-2~\cite{pan2024llmlingua2} 分别从预算控制、长上下文场景优化和数据蒸馏压缩模型等角度，探索如何在尽量保留关键信息的前提下降低 token 长度和推理延迟。这些方法的共同特点是：它们把"压缩"视为独立模块，而不是要求下游模型自己适应冗长输入；同时，它们通常同时报告压缩率、任务效果和推理速度，从而把成本收益明确纳入评价体系。

通用压缩工作的意义在于，它们为代码任务提供了重要方法学启发。首先，压缩并不等于粗暴截断，而是要在预算与保真之间进行权衡。其次，输入中的不同部分价值并不相同，因此预算分配和信息选择本身是关键问题。最后，压缩可能不仅降低成本，还可能通过去噪提高模型聚焦关键信息的能力。这些认识都与本文研究存在直接联系。

不过，通用压缩方法也存在明显局限。它们大多面向自然语言问答、摘要或一般长文场景，默认输入是近似线性的文本序列，而代码仓库往往具有显式层次结构、跨文件依赖和语法边界。若简单把代码视为普通文本进行 token 级压缩，很可能破坏结构线索，导致下游修复任务丢失关键上下文。因此，代码场景中的压缩不能完全照搬通用方法。

\subsection{面向代码的上下文压缩与选择}

针对代码结构特征，近年已出现更贴近代码任务的压缩与上下文管理研究。LongCodeZip 强调通用压缩方法难以保留代码依赖和结构信息，因此提出分层压缩与预算分配机制，从函数级和块级角度组织长代码上下文。更进一步，面向 issue resolution 的上下文压缩工作开始不再把目标定义为"保留一般语义"，而是直接定义为"保留修复所需 ingredients"。

算法~\ref{alg:context-compress} 给出了一种典型的面向仓库级修复的上下文压缩框架。

\begin{algorithm}[H]
\caption{面向仓库级修复的上下文压缩框架}
\label{alg:context-compress}
\KwIn{完整上下文序列 $C = [c_1, c_2, \ldots, c_n]$，token 预算 $B$，修复任务 $q$}
\KwOut{压缩后上下文 $C'$，满足 $|C'|_{\text{token}} \leq B$}
\tcc{阶段一：上下文分类}
将 $C$ 分类为三类：代码片段 $C_{code}$、搜索结果 $C_{search}$、历史轨迹 $C_{hist}$\;
\tcc{阶段二：重要性评分}
\For{每个 $c_i \in C$}{
  $\text{score}(c_i) \leftarrow w_1 \cdot \text{relevance}(c_i, q) + w_2 \cdot \text{recency}(c_i) - w_3 \cdot \text{redundancy}(c_i)$\;
}
\tcc{阶段三：分类预算分配}
$B_{code} \leftarrow \alpha B$；\quad $B_{search} \leftarrow \beta B$；\quad $B_{hist} \leftarrow (1{-}\alpha{-}\beta) B$\;
\tcc{阶段四：按预算从高分到低分贪心选取}
$C' \leftarrow \textsc{TopK}(C_{code}, B_{code}) \cup \textsc{TopK}(C_{search}, B_{search}) \cup \textsc{TopK}(C_{hist}, B_{hist})$\;
\Return $C'$\;
\end{algorithm}

针对 SWE-bench Verified 的压缩研究尝试通过蒸馏和任务驱动压缩策略，删去与修复无关的 observation 或代码片段，在降低 token 预算的同时保持甚至提升成功率。这些工作对本文尤其重要，因为它们说明代码修复中的压缩问题并不是一个抽象的长文本优化问题，而是与任务结构强耦合的过程优化问题。仓库级修复所需上下文往往既包括文件内容，也包括 issue 描述、搜索结果、测试输出和历史轨迹。如果只从单次 prompt 的角度理解压缩，就很难覆盖多轮 agent 场景中的上下文累积现象。

\subsection{前期压缩研究与本文的关系}

本文前期关于长上下文代码压缩的工作，可以作为当前毕设的重要研究基础，而不是孤立成果。该工作揭示了不同压缩策略在代码任务中的效果边界，说明代码结构、任务类型和压缩强度之间存在显著 trade-off。对当前论文而言，这一基础至少带来两方面价值。第一，它使本文能够避免把"压缩"简单理解为 token 删除，而更关注哪些信息对修复真正必要。第二，它为本文后续的方法设计提供了经验支点，即代码修复智能体中的成本优化必须建立在对仓库级上下文结构和任务充分性的理解之上。

换言之，前期长上下文压缩研究解决的是"代码长上下文该如何理解和处理"的基础问题，而本文进一步关注的是"当这些长上下文嵌入多轮修复智能体后，成本如何演化、该如何优化"。两者在问题链条上前后衔接，而不是重复。


\section{成本优化与效率分析相关研究}

\subsection{通用大模型系统中的成本优化}

随着大模型应用规模扩大，成本与效率优化很快成为独立研究主题。FrugalGPT~\cite{chen2023frugalgpt} 通过模型级联、提示适配等方式降低 API 调用成本，说明"更高质量系统"并不必然要求每次都使用最昂贵模型。随后，预算受限推理、时间预算推理和 token 预算控制等研究进一步把资源约束直接纳入模型决策过程。这些工作虽然并非针对代码智能体，但它们奠定了一个重要思想：资源消耗可以被视为一等设计目标，而不是系统完成后再附带统计的副产物。

对本文而言，通用成本优化研究的主要价值在于提供分析语言和设计视角。例如，预算不只是一个实验参数，还可以被显式纳入系统规划；延迟、token 和美元成本虽然形式不同，但都可以作为系统效率的外在表现；更重要的是，成本优化并不必然意味着效果退化，合理的选择和级联策略反而可能通过减少噪声提升系统稳定性。不过，这些工作大多仍聚焦单轮推理或少量交互场景，尚不足以解释多轮代码智能体中的轨迹累积成本。

\subsection{程序修复中的成本导向研究}

在 APR 方向，显式讨论成本的工作已经开始出现。CigaR~\cite{hidvegi2024cigar} 直接把 token 成本最小化作为优化目标之一，讨论如何在保持修复效果的前提下降低输入规模。ChatRepair~\cite{xia2023keep} 和 RepairAgent~\cite{bouzenia2024repairagent} 也都在论文中报告了单位 bug 的 token 或美元成本。这一点值得特别强调，因为它意味着程序修复研究已不再满足于"修好了多少"，而开始追问"修一次要付出多少代价"。

但从已有文献看，APR 成本研究仍主要停留在总量统计层面。多数工作会报告平均 token、平均美元成本或总体耗时，却较少深入分析这些成本到底来自哪些步骤。是问题定位最贵，还是编辑阶段最贵？失败样本为何常常消耗远高于成功样本的资源？多轮观察记录是如何在后续轮次中逐步放大成本的？这些问题在现有文献中通常没有得到细致分解。因此，虽然成本已经进入 APR 研究视野，但"成本归因"仍是相对缺失的。

\subsection{Agent Overhead、轨迹冗余与资源约束评估}

在 issue resolution agent 场景中，agent overhead 近两年受到越来越多关注。Agentless 从系统设计上指出，复杂 agent 脚手架未必天然优于更简洁的结构化流程，其背后隐含的一个重要原因就是复杂流程容易带来额外交互和额外成本。SWE-Effi 进一步提出，在资源约束下重新评价软件工程智能体的有效性，指出只比较 resolve rate 会掩盖大量昂贵失败样本，使榜单无法真实反映系统部署价值。AgentDiet 等轨迹级成本优化工作则直接把历史轨迹压缩、冗余 observation 删除等问题提到台前，说明多轮上下文本身就是重要成本来源。

这些研究共同说明，代码智能体的成本不是简单的"模型调用次数乘以平均价格"，而是与轨迹管理、失败重试和历史信息保留策略紧密相关。特别是在软件工程场景中，早期阅读和搜索产生的大量 observation 往往会在后续每一轮继续被拼入输入，从而形成明显的雪球效应。如果系统缺乏合理的历史管理机制，即使单次工具调用并不昂贵，累计成本也可能迅速失控。

\subsection{现有成本研究的不足}

尽管已有工作已经意识到成本问题的重要性，但总体上仍存在四方面不足。第一，成本常常仍被当作次要指标，而不是与任务能力并列的一等目标。第二，多数工作只统计总体 token 或美元开销，而缺少针对搜索、阅读、编辑、测试、重试等步骤的阶段化归因。第三，现有压缩研究多关注单次输入，却较少覆盖多轮 agent 过程中的动态上下文累积。第四，检索增强研究普遍重视"有没有找到相关信息"，却较少分析检索结果被写入 observation 后，对后续轮次输入长度的持续放大作用。

上述不足意味着，当前文献虽然已经承认"成本是问题"，但仍未充分回答"成本来自哪里"以及"究竟应该优先优化哪些部分"。这正是本文试图切入的研究空白。

\section{RLCoder：强化学习驱动的代码检索增强生成}

\subsection{方法概述}

RLCoder~\cite{wang2024rlcoder} 是一项将强化学习引入仓库级代码补全的代表性工作。传统检索增强代码生成（RAG for code）通常使用 BM25 或静态嵌入模型检索相关代码片段，但检索质量与生成质量之间存在天然的 gap——"语义相似"的候选未必是"生成有用"的候选。RLCoder 的核心思路是：用代码执行环境的反馈（编译通过率 + 测试通过率）作为强化学习奖励信号，直接优化检索器的排序策略，使检索结果真正有助于下游生成模型产生可执行的正确代码。

具体而言，RLCoder 由三个模块构成：\textbf{代码检索器}（Retriever）、\textbf{代码生成器}（Generator / LLM）和\textbf{执行环境}（Execution Environment）。检索器从仓库中选取 Top-$K$ 候选代码片段，生成器基于检索上下文补全目标代码，执行环境对补全结果进行编译和测试并返回量化奖励。强化学习训练器（PPO / REINFORCE）接收奖励后更新检索器参数，使其逐步学会选取对生成更有帮助的上下文片段。这一联合优化框架打破了传统 RAG 中检索器与生成器相互独立的假设，使两者在同一优化目标下协同演化。

RLCoder 的 RL 训练循环如算法~\ref{alg:rlcoder} 所示，
其系统架构如图~\ref{fig:rlcoder-arch}~所示。

\begin{algorithm}[H]
\caption{RLCoder: 强化学习驱动的检索增强代码生成训练}
\label{alg:rlcoder}
\KwIn{仓库 $\mathcal{R}$，补全任务集合 $\mathcal{D}$，奖励函数 $r(\cdot)$，最大迭代轮次 $T$}
\KwOut{优化后检索器参数 $\theta^*$}
初始化检索器策略 $\pi_\theta$（BM25 或预训练嵌入模型）\;
初始化生成器 $\mathcal{M}$（预训练代码 LLM，参数冻结）\;
\For{$t = 1, \ldots, T$}{
  \For{每个补全任务 $(q, y^*) \in \mathcal{D}$}{
    \tcc{检索阶段：策略 $\pi_\theta$ 从仓库中采样 Top-K 候选}
    $\mathcal{C} \leftarrow \pi_\theta(q,\;\mathcal{R})$\;
    \tcc{生成阶段：LLM 基于检索上下文生成补全}
    $\hat{y} \leftarrow \mathcal{M}(q,\;\mathcal{C})$\;
    \tcc{奖励计算：执行环境评估生成结果}
    $r_t \leftarrow r(\hat{y},\; y^*)$\tcp*{编译通过率 + 测试通过率}
    \tcc{策略梯度更新（REINFORCE / PPO）}
    $\nabla_\theta \mathcal{J} \leftarrow r_t \cdot \nabla_\theta \log \pi_\theta(\mathcal{C} \mid q)$\;
    $\theta \leftarrow \theta + \alpha \nabla_\theta \mathcal{J}$\;
  }
}
\Return $\theta^* \leftarrow \theta$\;
\end{algorithm}

\subsection{RLCoder 与本文研究的关系}

从本文视角看，RLCoder 具有两方面重要参考价值。

第一，\textbf{成本视角}：RLCoder 的 RL 训练阶段需要大量与执行环境的交互，会产生可观的训练期成本；但一旦训练完成，优化后的检索器能够在推理阶段更准确地定位相关上下文，从而减少生成器在错误上下文上的无效推理消耗。这说明"离线投入—在线节省"的成本结构在代码修复场景中同样适用，与本文关注的推理阶段成本优化方向形成互补。

第二，\textbf{方法视角}：RLCoder 通过环境反馈信号优化了修复流程中的检索阶段，证明了将执行结果作为训练信号可以显著改善上下文质量。这一思想启示本文：在多轮修复智能体的成本分析中，不仅要关注"哪个阶段产生了多少 token"，还应关注"哪个阶段的输出质量直接影响后续阶段的重试次数和累积成本"——检索质量差会导致更多修复轮次，从而在整体轨迹层面放大成本。


\section{现有研究不足与本文切入点}

\subsection{现有研究的系统性不足}

在自动程序修复方向，传统 APR、学习式 APR 和 LLM-based APR 已经为补丁生成问题积累了丰富方法，但绝大多数工作更关注修复成功与否，而非完整修复过程的资源代价。即便是已经报告 token 或美元成本的工作，也往往停留在总体统计层面，缺少对失败样本、冗余 observation 和阶段性成本来源的系统分析。

在软件工程智能体方向，已有系统已经非常重视环境交互、工具调用和多轮反馈，但大多数论文仍把成功率与 benchmark 排名放在首位，对 agent overhead 的讨论通常是局部的、补充性的。更具体地说，研究者已经开始意识到复杂流程会产生额外开销，却还没有形成一套足够统一的分析框架，将定位、检索、阅读、编辑、测试和历史累积这些阶段纳入同一成本视角。

在长上下文代码理解与压缩方向，现有工作已经证明简单扩展窗口并不能自动解决问题，压缩与选择策略也已表现出明显潜力。然而，这些研究大多面向单次任务输入，对多轮智能体轨迹的动态演化覆盖不足。仓库级修复中的历史 observation 不只是长文本，它还是会被不断改写、重复引用和跨轮叠加的过程性上下文，这与普通长文问答中的压缩问题存在本质差别。

在 benchmark 和任务场景方向，SWE-bench 及其衍生工作已经把研究推进到更真实的软件维护环境，但随着任务越来越真实，实验和部署成本也同步上升。若没有针对成本的专门分析，仅比较最终成功率，就很难充分回答系统在现实环境中是否可用、是否值得部署。

\subsection{本文的研究基础}

本文的研究并非脱离已有成果单独展开，而是建立在两类前期工作之上。第一类是长上下文代码压缩研究。前期工作已经对代码任务中的压缩方法与效果边界进行了系统探索，积累了关于压缩率、任务充分性和结构敏感性的经验认识。这一基础使本文能够更有针对性地讨论：哪些上下文是修复过程真正需要保留的，哪些内容只是造成后续轮次负担的噪声。

第二类是 GitHub issue resolution 基准扩展研究。OmniGIRL~\cite{guo2025omnigirlmultilingualmultimodalbenchmark} 作为多语言、多模态 issue resolution benchmark，说明真实软件维护任务远比单语言、单模态的简化场景复杂，也提示本文不能把代码修复智能体的成本问题仅理解为单一仓库或单一语言上的偶发现象。它为本文提供了更真实的任务背景和更宽的场景视角。

将这两类前期工作结合起来，可以形成一条清晰的问题链：长上下文代码任务告诉我们"上下文本身难以有效利用"，issue resolution 基准扩展告诉我们"真实软件任务的复杂性仍在持续增加"，而当前毕设则进一步追问"当这些复杂上下文被放入多轮修复智能体时，成本是如何产生并被放大的"。因此，前期工作并不是独立成果堆砌，而是当前论文主线的直接研究基础。

\subsection{本文的研究切入点}

基于上述文献综述与研究基础，本文的切入点可以概括为四个方面。第一，本文将代码修复智能体的推理成本视为独立研究对象，而不把它当作成功率分析后的附带结果。第二，本文关注的是过程性成本而非单次 prompt 成本，即把完整修复轨迹中的搜索、阅读、编辑、验证和重试都纳入分析范围。第三，本文强调成本归因的阶段化视角，希望回答"哪里最贵、为什么最贵、哪些步骤最值得优先优化"。第四，本文把成本与能力放在联合评价框架中讨论，而不是单独追求最低 token 或最高 resolve rate。

据此，本文与现有工作的区别不在于简单重复某个 benchmark 或压缩方法，而在于试图把 APR、软件工程智能体、长上下文代码理解和 issue resolution 这几条原本相对分散的研究线索收束到同一个工程问题上：如何理解并优化代码修复智能体在真实仓库级任务中的高推理成本。

\section{本章小结}

本章围绕本文主题，对自动程序修复、软件工程智能体、仓库级代码任务、长上下文代码理解与压缩，以及成本优化与效率分析等方向进行了系统梳理，并重点介绍了 RLCoder 这一将强化学习引入代码检索增强生成的代表性工作。文献回顾表明，程序修复研究已从传统搜索和模板方法演化到大语言模型驱动与交互式修复；软件工程智能体研究已从单轮生成发展到多轮工具调用与环境交互；代码任务基准已从函数级问题推进到真实 GitHub issue resolution；而长上下文压缩与成本优化研究则逐渐揭示出"上下文越长、任务越真实、成本越难控制"的共同趋势。

更重要的是，现有研究虽然已经分别讨论了修复能力、仓库理解、上下文压缩和资源约束，但对代码修复智能体过程性成本的系统刻画仍然不足，尤其缺少针对不同执行阶段的成本归因分析。本文正是在这一研究空白上展开：以前期长上下文代码压缩研究和 issue resolution benchmark 扩展研究为基础，进一步把焦点落在真实仓库级修复中的高推理成本问题上。下一章将据此对代码修复智能体的任务流程、成本定义和分析框架进行形式化说明。
