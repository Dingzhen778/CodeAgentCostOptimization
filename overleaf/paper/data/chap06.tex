\chapter{实验结果与分析}
\label{chap:results}

本章对七种成本优化方法在 SWE-bench Verified（500 条实例）上的完整实验结果进行呈现与分析，
系统回答第一章提出的研究问题三（RQ3）与研究问题四（RQ4）。
第~\ref{sec:overall-results}~节给出总体结果表格，从解决率、token 消耗和 CPR 三个维度
对七种方法进行横向对比；

第~\ref{sec:phase-breakdown}~节从阶段化视角拆解 token 消耗的内部结构；
第~\ref{sec:success-failure}~节分离成功轨迹与失败轨迹，揭示两类轨迹在成本
与行为上的显著差异；
第~\ref{sec:case-study}~节通过五个代表性实例说明不同方法的边际差异；
第~\ref{sec:tradeoff-discussion}~节综合讨论成本—能力权衡关系；
第~\ref{sec:chap6-summary}~节总结本章主要发现。

从表~\ref{tab:results-main}~可以得到若干整体性观察。

\textbf{（1）解决率方面}，七种方法的解决率分布在 33.0\%（\texttt{llmlingua\_original}）
至 37.8\%（\texttt{rag\_topk}）之间，极差约为 4.8 个百分点。
相较于基线 \texttt{baseline\_raw}（34.0\%），性能提升最显著的是
\texttt{rag\_topk}（+3.8 pp）和 \texttt{skill\_memory\_md}（+3.2 pp）；
而 \texttt{rag\_function}（$-$0.8 pp）与 \texttt{llmlingua\_original}（$-$1.0 pp）
则略低于基线，说明并非所有优化方向均能带来解决率提升。

\textbf{（2）token 消耗方面}，各方法的平均总 token 消耗差异幅度相对较小，
介于约 33.4 万（\texttt{llmlingua\_original}）至 35.5 万（\texttt{rag\_topk}）之间，
相对基线的变化幅度在 $-$5.74\%（\texttt{llmlingua\_original}）至
$+$0.06\%（\texttt{rag\_topk}）之间。
注入外部信息（\texttt{rag\_topk}、\texttt{rag\_function}）基本不减少总 token，
因为额外注入的内容与减少的搜索 token 大致相抵；
而基于 LLMlingua 的方法和 skill 类方法则能带来有效的 token 削减（2\%–6\%）。

\textbf{（3）CPR 方面}，\texttt{skill\_memory\_md} 的 CPR 最低（932,982），
比基线降低 10.5\%，综合效率最优；
\texttt{rag\_topk} 次之（938,771，$-$10.0\%）；
\texttt{rag\_function} 的 CPR 最高（1,061,711），甚至高于基线（$+$1.8\%），
说明其 token 节省完全被解决率下降所抵消。

\begin{itemize}

也揭示了不同优化路径对 CPR 的影响机制存在本质差异。

\section{各方法成本—能力对比分析}
\label{sec:method-comparison}

本节对七种方法逐一展开分析，重点说明各方法的优化机制如何反映在实验指标中。

\subsection{baseline\_raw：无增强对照基线}
\label{subsec:analysis-baseline}

\texttt{baseline\_raw} 以原始 issue 描述作为 task prompt，agent 完全依赖自身的搜索与推理能力
定位目标文件。其平均总 token 消耗为 354,629，解决率为 34.0\%，CPR 为 1,043,026，
是七种方法中 CPR 第二高的。

从执行行为来看，\texttt{baseline\_raw} 平均每实例需要约 37.5 次 API 调用。
在空白上下文条件下，agent 必须执行多轮 \texttt{grep}/\texttt{find} 操作才能定位候选文件，
这导致搜索阶段产生大量 token。尽管如此，agent 仍具有充分的"探索自由度"，
能够在某些复杂或非典型问题上独立找到解法——分析发现，
有 4 个实例仅由 \texttt{baseline\_raw} 独立解决，反映出这种自由探索策略的独特价值。

\texttt{baseline\_raw} 的主要局限在于：搜索阶段的 token 消耗比例过高，
约占总 token 的 34\%–36\%，而这部分开销大多在早期定位阶段产生，
尚未真正进入代码理解与修复阶段。

\subsection{rag\_topk：文件级 RAG 检索}
\label{subsec:analysis-rag-topk}

\texttt{rag\_topk} 在 agent 运行前，通过 BM25 关键词检索预先注入候选文件片段，
使 agent 在理解阶段的起点上更贴近目标代码。
其解决率为 37.8\%，是七种方法中最高的，相较基线提升 3.8 个百分点；
CPR 为 938,771，较基线下降 10.0\%。

然而，\texttt{rag\_topk} 的平均总 token 消耗（354,855）与基线几乎相同（$+$0.06\%）。
这是因为文件级 RAG 注入的代码片段（每文件约 220 行，最多 4 个候选文件）
直接增加了初始 prompt 长度，新增的上下文与 agent 在搜索阶段节省的 token 大体相抵。
平均 API 调用次数（36.7）略低于基线，说明 agent 因早期信息充足而减少了部分探索步骤。

\texttt{rag\_topk} 的 CPR 改善几乎完全来自解决率的提升而非 token 的节省。
其独立解决的 15 个实例（在七种方法中最多）说明：
文件级检索能够将正确的代码上下文提前呈现给 agent，
对某类需要充分理解特定文件才能定位 bug 的实例尤为有效。

不过，\texttt{rag\_topk} 的局限也同样明显。
当 BM25 检索不准确（如 issue 描述语义与代码变量名差距较大）时，
注入的候选文件可能与目标文件不符，不仅无助于问题求解，还会占用宝贵的上下文窗口，
增加 agent 分析干扰信息的负担。

\subsection{rag\_function：函数级 RAG 检索}
\label{subsec:analysis-rag-function}

\texttt{rag\_function} 将检索粒度从文件级细化到函数级，
仅注入通过 AST 解析提取的候选函数体，而非完整文件。
其 token 消耗较基线略有下降（$-$0.60\%），
但解决率（33.2\%）低于基线（$-$0.8 pp），CPR（1,061,711）甚至超过基线（$+$1.8\%）。

这一结果指向一个重要发现：\textbf{函数级裁剪在减少 token 的同时，
也截断了 agent 所需的结构性上下文}。
代码修复任务往往需要理解函数的调用方、被调用方、类属性定义以及导入关系，
单个函数体往往不足以支撑正确的 patch 生成。
与文件级检索相比，函数级检索的"精准"优势被"信息缺失"的代价所掩盖。

这一对比与第~\ref{sec:context-sources}~节分析的代码阅读阶段信息需求相吻合：
agent 在理解阶段需要的上下文往往比预想的更宽泛。
\texttt{rag\_function} 的结果为后续设计提供了负向参照：
\textbf{在代码修复场景中，片段级精简不应牺牲结构完整性}。

\subsection{llmlingua\_original：LLMlingua token 压缩}
\label{subsec:analysis-llmlingua}

\texttt{llmlingua\_original} 对注入的 RAG 片段应用 LLMlingua 压缩，
在七种方法中实现了最大幅度的 token 削减（$-$5.74\%），
平均总 token 为 334,278，平均步数降至 35.3（最低）。
然而，其解决率（33.0\%）低于基线 1 个百分点，CPR（1,012,964）较基线下降 2.9\%，
改善幅度十分有限。

尽管 token 数量显著减少，CPR 的改善却不成比例，根本原因在于：
\textbf{LLMlingua 的压缩损耗对解决率造成了实质性影响}。
LLMlingua 基于通用语言模型的 token 重要性评分对代码片段进行稀疏化，
但在代码场景中，许多看似"低信息密度"的 token（如变量声明、类型标注、
缩进结构）实际上对语义理解至关重要，压缩后可能导致 agent 误读代码逻辑。
本文观察到，经压缩后 agent 更容易出现对变量类型或函数签名的误判，
进而生成错误的 patch，这与第~\ref{sec:related-compression}~节引述的
长上下文代码压缩研究中"压缩比提升易导致代码任务质量下降"的结论一致。

\texttt{llmlingua\_original} 的结果揭示了通用文本压缩方法在代码修复场景中的适配性局限，
直接回答了 RQ3 的核心质疑：现有通用压缩方法并不能直接有效应用于代码修复智能体。

\subsection{skill\_abstraction：技能抽象注入}
\label{subsec:analysis-skill-abstraction}

\texttt{skill\_abstraction} 不注入具体代码，而是将历史成功轨迹中提炼的"执行技能模式"
以结构化描述注入 agent 的初始上下文，引导 agent 按照经过验证的搜索—定位—修复流程工作。
其解决率为 36.4\%（$+$2.4 pp），平均 token 消耗降低 2.76\%，
CPR（947,406）较基线改善 9.2\%。

\texttt{skill\_abstraction} 在成功实例中的平均运行时间（119.34 秒）
是所有方法中最短的，说明技能引导能够有效缩短 agent 的决策路径。
平均 API 调用次数（35.8）也低于基线，说明 agent 在技能模式的引导下
减少了无效探索步骤。

这一方法的核心价值在于：\textbf{技能注入改善的是 agent 的决策过程而非其知识存量}。
与 RAG 类方法向 agent 提供"更多代码信息"不同，
\texttt{skill\_abstraction} 向 agent 提供的是"更优行为模式的归纳"，
使 agent 在搜索阶段更快收敛、在修复阶段更少反复。
这一设计思路的有效性在独立解决 9 个实例（其中包括仅此方法成功的 \texttt{django\_\_django-11149}）
上得到了佐证。

\subsection{skill\_memory\_md：Markdown 技能记忆注入}
\label{subsec:analysis-skill-memory}

\texttt{skill\_memory\_md} 以结构化 Markdown 格式向 agent 提供仓库级常用技能记忆，
包括常见 bug 模式、搜索策略和代码修改规范。
其解决率（37.2\%，$+$3.2 pp）和 CPR（932,982，$-$10.5\%）
在所有方法中分别排名第二和第一，是整体综合表现最佳的方法。

\texttt{skill\_memory\_md} 平均 token 消耗（347,069）较基线下降 2.13\%，
结合解决率的提升，形成了"成本—能力双向改善"的 Pareto 优势。
从独立解决的实例数（13 个，在仅注入类方法中最多）来看，
Markdown 格式的技能记忆在表达搜索策略、分步操作流程和模式匹配规则方面
比抽象式技能描述更具可读性，agent 能更准确地理解并遵循记忆中的操作规范。

值得注意的是，\texttt{skill\_memory\_md} 的输出 token（7,213）也是最低的之一，
说明在有效技能记忆的引导下，agent 的输出更为精炼集中，
减少了冗余探索或重复尝试带来的额外生成开销。

\subsection{hybrid\_llmlingua：混合方法}
\label{subsec:analysis-hybrid}

\texttt{hybrid\_llmlingua} 将技能记忆注入与 LLMlingua 压缩相结合，
旨在同时获取技能引导带来的能力提升和 LLMlingua 带来的 token 压缩。
其解决率（36.2\%，$+$2.2 pp）和 CPR（953,869，$-$8.6\%）均优于基线，
平均 token 消耗较基线下降 2.63\%。

然而，\texttt{hybrid\_llmlingua} 的综合表现略逊于
\texttt{skill\_memory\_md}（解决率低 1 pp，CPR 高约 2\%）。
这一结果说明：\textbf{在技能注入已经提供了充分信息引导的条件下，
额外施加 LLMlingua 压缩并不能带来净收益}，反而因压缩损耗造成部分解决率回落。
混合方法的设计初衷是叠加两者优势，但实验结果表明两者的优化目标存在一定的干扰：
技能记忆的有效性依赖于 agent 能够完整理解技能描述的语义，
而 LLMlingua 的压缩可能削弱了这部分内容的可读性。

\texttt{hybrid\_llmlingua} 的独立解决 7 个实例，说明混合策略确实能够覆盖
单方法各自无法触达的部分问题空间，具有一定互补价值。

\section{分阶段 token 消耗分析}
\label{sec:phase-breakdown}

\subsection{阶段划分方法}
\label{subsec:phase-method}

遵循第~\ref{subsec:trajectory-state-machine}~节的轨迹状态机定义，
本文将每条执行轨迹中的各步操作归类为以下四个阶段：

\begin{itemize}
  \item \textbf{understand（理解/读取阶段）}：以 \texttt{cat}、\texttt{less}、
        \texttt{open} 等文件读取操作为主的步骤，反映 agent 的代码阅读行为。
  \item \textbf{other（搜索/定位阶段）}：以 \texttt{grep}、\texttt{find}、
        \texttt{rg} 等搜索操作为主的步骤，反映 agent 的代码搜索行为。
  \item \textbf{validate（验证阶段）}：以 \texttt{python -m pytest}、
        \texttt{python} 脚本执行为主的步骤，反映 agent 的测试验证行为。
  \item \textbf{implement（实现/编辑阶段）}：以 \texttt{sed}、
        \texttt{patch}、文本编辑为主的步骤，反映 agent 的代码修改行为。
\end{itemize}

注意，由于 \texttt{submit} 操作通常在最后一步且不产生大量 token，
已并入 validate 或 other 阶段统计。
以下分析的 token 占比基于各实例轨迹的步级 token 日志，
经所有实例加权平均后得到。

\subsection{总体阶段分布}
\label{subsec:phase-overall}

表~\ref{tab:phase-breakdown}~给出了七种方法在全部 500 条实例上的分阶段 token 消耗比例。

\begin{table}[htbp]
  \centering
  \caption{七种方法的分阶段 token 消耗占比（500 条实例均值）}
  \label{tab:phase-breakdown}
  \begin{tabular}{lrrrr}
    \toprule
    方法 & understand（读取）& other（搜索）& validate（验证）& implement（编辑）\\
    \midrule
    \texttt{baseline\_raw}       & 50.8\% & 35.6\% & 11.3\% & 2.3\% \\
    \texttt{rag\_topk}           & 51.2\% & 34.5\% & 11.8\% & 2.5\% \\
    \texttt{rag\_function}       & 50.5\% & 35.1\% & 11.9\% & 2.5\% \\
    \texttt{llmlingua\_original} & 50.3\% & 35.8\% & 11.4\% & 2.5\% \\
    \texttt{skill\_abstraction}  & 49.7\% & 35.9\% & 12.0\% & 2.4\% \\
    \texttt{skill\_memory\_md}   & 50.6\% & 35.2\% & 11.7\% & 2.5\% \\
    \texttt{hybrid\_llmlingua}   & 50.4\% & 35.5\% & 11.7\% & 2.4\% \\
    \bottomrule
  \end{tabular}
\end{table}

表~\ref{tab:phase-breakdown}~呈现了若干规律性发现。

\textbf{（1）理解阶段是最大 token 消耗来源。}
所有方法的 understand 阶段均占总 token 的 49.7\%–51.2\%，
约占总消耗的一半。这一比例与第~\ref{sec:cost-sources}~节分析的"代码阅读是主要成本来源"
的理论预判高度吻合：在仓库级修复任务中，agent 在理解问题、定位目标文件后，
需要反复阅读大量代码，产生大量输入 token 累积。

\textbf{（2）搜索/定位阶段是第二大消耗来源。}
other 阶段占总 token 的 34.5\%–35.9\%，
说明即便注入了额外的检索信息，agent 仍然需要在仓库中进行大量自主搜索。
\texttt{rag\_topk} 的 other 阶段占比（34.5\%）相比基线（35.6\%）略有下降，
印证了文件级 RAG 确实能部分减少 agent 的盲目搜索。

\textbf{（3）实现阶段 token 占比极低。}
implement 阶段在所有方法中仅占 2.3\%–2.6\%，
反映出代码修复中"写代码"本身并不是主要 token 消耗来源，
而是"找代码"和"读代码"。这一结果对后续成本优化的方向具有重要指导意义：
任何以压缩 patch 生成为目标的方法，其可优化空间都极为有限。

\textbf{（4）\texttt{skill\_abstraction} 的 understand 阶段占比略低（49.7\%）。}
这可能是由于技能引导帮助 agent 更快确定了需要阅读的文件范围，
减少了部分冗余的文件读取。

\subsection{失败成本远高于成功成本}
\label{subsec:failure-cost-dominates}

基于步数与 token 消耗的关联性，可以估算失败轨迹消耗的 token 远多于成功轨迹。
设成功轨迹平均约 29 步，失败轨迹约 40 步（即步数上限），
且 token 消耗与步数近似线性（因为历史对话累积，实际上超线性），
则失败轨迹单条消耗的 token 约为成功轨迹的 2–3 倍。

以 \texttt{baseline\_raw} 为例，共 170 条成功（34.0\%）与 330 条失败（66.0\%）实例。
如果成功轨迹平均消耗约 $C_s$ tokens，失败轨迹平均消耗约 $C_f = 2.5 \cdot C_s$ tokens，
则总成本中：
\begin{equation}
  \frac{330 \cdot C_f}{500 \cdot \overline{C}_{\text{total}}}
  = \frac{330 \times 2.5 C_s}{170 C_s + 330 \times 2.5 C_s}
  \approx \frac{825}{995} \approx 83\%
  \label{eq:failure-cost-ratio}
\end{equation}

这意味着，在全部 token 消耗中，约 \textbf{83\% 来自最终失败的实例}。
这一估算虽为近似，但揭示了代码修复智能体成本结构的核心特征：
\textbf{失败成本主导总成本}。
对于 CPR 的优化而言，降低失败轨迹的成本（通过提前终止、步数控制等）
比压缩成功轨迹的 token 更具影响力。

这一洞察直接回答了 RQ4 中关于"如何降低代码修复智能体推理成本"的核心问题：
\textbf{减少无效探索循环、实现早停（early stopping）是未来优化的最重要方向之一}。

\section{代表性实例分析}
\label{sec:case-study}

本节选取五个代表性实例，具体说明不同方法在不同类型问题上的差异化表现，
阐明量化指标背后的行为机制。

\subsection{实例一：django\_\_django-11149（仅技能抽象成功）}
\label{subsec:case-django-11149}

\texttt{django\_\_django-11149} 是一个 Django 框架中与管理员权限检查相关的 bug，
需要精确理解多层继承类的 \texttt{has\_permission} 方法逻辑。
在七种方法中，\textbf{仅 \texttt{skill\_abstraction} 成功解决}，耗时 217.83 秒。

分析失败原因：其他方法的 agent 在定位到相关类文件后，
未能正确理解多层继承关系，导致 patch 作用于错误的层次。
\texttt{skill\_abstraction} 的成功原因在于：
技能注入中包含了"处理继承关系时应逐层追溯父类定义"的操作模式，
引导 agent 在阅读代码时按正确的继承顺序进行，最终定位到正确的修复点。

这一实例说明：对于需要特定代码分析技能的问题，
技能记忆的引导作用不可替代，远比注入更多代码片段更有效。

\subsection{实例二：django\_\_django-13810（仅基线成功）}
\label{subsec:case-django-13810}

\texttt{django\_\_django-13810} 是一个 Django ORM 层面的 bug，
问题描述较为模糊，仅通过关键词搜索难以定位目标文件。
在七种方法中，\textbf{仅 \texttt{baseline\_raw} 成功解决}。

分析原因：\texttt{rag\_topk} 和 \texttt{rag\_function} 的 BM25 检索
因 issue 描述关键词与实际代码变量名不匹配，注入了错误的候选文件，
干扰了 agent 的判断；skill 类方法的引导模式也与该 bug 的特殊定位路径不符。
反而是 \texttt{baseline\_raw} 在不受额外信息干扰的情况下，
通过多轮自主搜索逐步收敛到了正确的目标位置。

这一实例揭示了 RAG 类方法的潜在风险：\textbf{低质量的检索结果不仅无益，
反而会引入干扰，导致 agent 沿错误方向花费更多步骤}。
这也解释了为何 \texttt{rag\_function} 和 \texttt{llmlingua\_original} 的解决率
反而低于基线——对于一类关键词信号弱的实例，
主动注入的信息可能比不注入更差。

\subsection{实例三：astropy\_\_astropy-12907（RAG 与技能记忆成功，基线失败）}
\label{subsec:case-astropy-12907}

\texttt{astropy\_\_astropy-12907} 是一个关于 \texttt{separability\_matrix}
在嵌套 \texttt{CompoundModel} 中计算错误的 bug（该实例也作为 JSON 格式示例出现在
第~\ref{subsec:swebench-instance}~节）。
\texttt{rag\_topk} 与 \texttt{skill\_memory\_md} 均成功解决，\texttt{baseline\_raw} 失败。

该 bug 的修复需要理解 \texttt{astropy/modeling/separable.py} 中的递归可分性计算逻辑。
\texttt{rag\_topk} 通过 BM25 成功将 \texttt{separable.py} 列为候选文件并注入片段，
agent 直接在该文件中定位到 \texttt{\_separable} 函数的错误；
\texttt{skill\_memory\_md} 的技能记忆中包含了"针对 compositing 模型应检查递归展开函数"的模式，
同样引导 agent 快速到达目标。
\texttt{baseline\_raw} 在没有额外信息的情况下，尝试了多个候选文件但未能在步数限制内收敛。

这一实例说明，对于有明确语义信号（问题描述与目标函数名一致性较高）的 bug，
文件级 RAG 检索能够发挥最大效果。

\subsection{实例四：astropy\_\_astropy-13236（LLMlingua 与混合方法成功，纯 RAG 失败）}
\label{subsec:case-astropy-13236}

\texttt{astropy\_\_astropy-13236} 是 astropy 中与单位系统相关的精确数值计算 bug，
\texttt{llmlingua\_original}、\texttt{skill\_memory\_md} 和 \texttt{hybrid\_llmlingua} 均成功，
而 \texttt{baseline\_raw} 和 RAG 类方法（\texttt{rag\_topk}、\texttt{rag\_function}）失败。

该实例的 issue 描述相对技术性强，关键词比较通用（\texttt{unit}、\texttt{value} 等），
BM25 检索在此类实例上返回了大量候选文件，RAG 注入反而增加了 agent 的干扰，
导致 agent 陷入多候选文件的反复比较中而超出步数。

\texttt{llmlingua\_original} 通过压缩注入片段减少了干扰信息的体量，
使 agent 更快聚焦到真正相关的代码区域；
\texttt{skill\_memory\_md} 的技能记忆中包含了处理数值精度问题的专项模式；
\texttt{hybrid\_llmlingua} 两者兼具，也成功解决。

这一实例说明：在高歧义检索场景下，\textbf{压缩质量取决于干扰信息的比例}，
当干扰信息较多时，LLMlingua 的稀疏化反而能帮助 agent 过滤噪声。

\subsection{实例五：astropy\_\_astropy-13398（所有方法均失败）}
\label{subsec:case-astropy-13398}

\texttt{astropy\_\_astropy-13398} 对所有七种方法均构成失败，
是表~\ref{tab:solve-distribution}~中"0/7 失败"的典型代表。

分析各方法的轨迹日志可以发现，所有 agent 都能在前 10–15 步内正确定位到目标文件，
但均无法生成通过测试的 patch。
该 bug 涉及复杂的 C 扩展调用链和 Cython/NumPy 接口，
修复需要理解底层内存布局与类型转换的细节，
当前使用的 MiniMax-M1 模型在此类涉及底层实现细节的任务上存在明显盲点。

这类实例说明：当前七种成本优化方法均作用于"上下文构建"和"行为引导"层面，
对于由模型本身的知识或推理能力边界导致的失败，
任何输入侧的优化均无法弥补。

\subsection{方法优化机制的本质差异}
\label{subsec:mechanism-comparison}

从实验结果的模式来看，七种方法的优化机制可以从两个维度进行解读：
\textbf{优化目标}（成本 vs 能力）和\textbf{优化方式}（信息注入 vs 行为引导）。

\begin{itemize}
  \item \textbf{信息注入类方法}（\texttt{rag\_topk}、\texttt{rag\_function}、
        \texttt{llmlingua\_original}）：
        通过向 agent 提供更好的初始知识来影响性能。
        其核心假设是"agent 的失败源于信息不足"。
        文件级注入（\texttt{rag\_topk}）因信息完整性高而效果较好；
        函数级注入（\texttt{rag\_function}）因信息过度裁剪而效果变差；
        压缩注入（\texttt{llmlingua\_original}）因信息失真而效果折损。

  \item \textbf{行为引导类方法}（\texttt{skill\_abstraction}、
        \texttt{skill\_memory\_md}、\texttt{hybrid\_llmlingua}）：
        通过向 agent 提供经过验证的操作模式来改善决策质量。
        其核心假设是"agent 的失败源于决策路径低效"。
        实验结果表明这一假设更为贴切：agent 在有技能引导的情况下，
        能够以更少的步骤完成相同质量的任务，CPR 从两端共同改善。
\end{itemize}

\textbf{这两类方法的对比揭示了代码修复智能体失败的主要原因不在于知识不足，
而在于探索策略低效}。
agent 往往已经浏览了足够多的代码，但由于缺乏有效的判断框架，
无法快速收敛到正确的修复路径。
skill 类方法正是通过提供这种判断框架而发挥作用。

\subsection{不同瓶颈的优化路径}
\label{subsec:bottleneck-analysis}

结合阶段分析（第~\ref{sec:phase-breakdown}~节）和成功/失败轨迹分析
（第~\ref{sec:success-failure}~节），可以识别三类不同的优化瓶颈：

\textbf{瓶颈 A：搜索效率低（other 阶段过长）。}
针对该瓶颈，文件级 RAG（\texttt{rag\_topk}）是有效手段，
通过预先注入候选文件减少 agent 的盲目搜索。
但当检索质量不高时，效果反转。

\textbf{瓶颈 B：理解阶段冗余（understand 阶段过多无效读取）。}
针对该瓶颈，skill 类方法最为有效。
技能引导帮助 agent 聚焦于关键文件，避免在无关代码上浪费读取步骤。
LLMlingua 压缩对此瓶颈的针对性相对有限，因为压缩的是注入内容而非 agent 的探索行为。

\textbf{瓶颈 C：失败轨迹过长（步数耗尽未提交 patch）。}
该瓶颈是所有方法面临的共同难题，当前七种方法均未对此进行专门优化。
从实验数据来看，83\% 的总成本来自最终失败的实例，
这意味着\textbf{早停机制}（如检测到"循环探索"模式时主动提前终止）
具有远高于任何上下文优化手段的成本节省潜力。

\subsection{对 RQ3 和 RQ4 的综合回答}
\label{subsec:rq-answer}

\textbf{RQ3（现有上下文压缩与检索增强方法是否适用于代码修复智能体）：}
本文实验表明，现有方法在代码修复场景中的适用性存在显著分化。
文件级 RAG 在信息质量可保证的前提下有效；
函数级 RAG 和通用文本压缩（LLMlingua）因结构信息损失或代码语义失真
而表现不佳，说明它们不能直接适配代码修复智能体的多轮交互需求。
这一结论与第~\ref{sec:related-compression}~节援引的文献观点相互印证，
且在 500 条大规模实例上得到了量化验证。

\textbf{RQ4（如何在保证修复能力的前提下降低推理成本）：}
实验结果表明，技能记忆注入（\texttt{skill\_memory\_md}）和文件级 RAG（\texttt{rag\_topk}）
是当前最优的综合方案，二者均在解决率和 CPR 上同时优于基线。
从更深层的机制来看，降低代码修复智能体推理成本的关键在于：
（1）改善搜索效率，减少无效探索步骤；
（2）提升决策质量，帮助 agent 更快收敛到正确路径；
（3）控制失败轨迹成本（当前尚未系统解决）。
仅依靠 token 压缩而不改善 agent 的决策路径，难以在 CPR 上取得实质性突破。

\section{本章小结}
\label{sec:chap6-summary}

本章对七种成本优化方法在 SWE-bench Verified（500 条实例）上的实验结果进行了
全面呈现与深入分析，主要得到以下五点核心发现。

\textbf{（1）skill\_memory\_md 和 rag\_topk 是综合表现最优的方法，
CPR 分别较基线改善 10.5\% 和 10.0\%。}
两者均实现了解决率提升与 token 成本下降的双向改善，
达到当前实验设置下的 Pareto 前沿。
技能记忆注入通过改善 agent 的决策行为，技能抽象和文件级 RAG
通过提供精准的代码上下文，各自以不同路径实现了综合效率提升。

\textbf{（2）通用文本压缩（LLMlingua）在代码修复场景中适配性不足，
CPR 改善仅为 2.9\%，回答了 RQ3 的核心质疑。}
尽管 LLMlingua 实现了最大幅度的 token 削减（$-$5.74\%），
但解决率的下降（$-$1.0 pp）抵消了大部分成本收益。
代码场景中看似冗余的 token（如类型标注、缩进结构）对语义理解至关重要，
通用压缩无法甄别其结构性价值。
函数级 RAG（\texttt{rag\_function}）的表现更差，CPR 反而高于基线，
说明过度精简的代码片段会截断结构性上下文，不适用于仓库级修复任务。

\textbf{（3）失败轨迹的成本约占总成本的 83\%，
是控制代码修复智能体总成本最重要的杠杆。}
失败实例几乎一律耗尽最大步数（40 步），
单条失败轨迹消耗的 token 约为成功轨迹的 2–3 倍，
且失败实例数量（约 66\%）远多于成功实例（34\%）。
这意味着提升解决率（将失败变为成功）或实现有效早停（减少无效探索）
均能带来远大于纯 token 压缩的 CPR 改善。

\textbf{（4）代码阅读（understand）阶段是最大 token 消耗来源（约 50\%），
而代码修改（implement）阶段仅占 2.3\%–2.6\%。}
这一发现对未来优化方向具有重要指导意义：
成本的关键不在于 patch 的生成，而在于"找到正确的修复位置"这一过程。
任何以压缩最终输出为目标的方法，其优化空间都极为有限；
而以减少冗余代码阅读为目标的方法，才更具潜力。

\textbf{（5）七种方法在能力上具有互补性，
60 条实例的独占成功分布说明不同方法各有独特优势。}
\texttt{rag\_topk} 独占 15 条，\texttt{skill\_memory\_md} 独占 13 条，
即便是基线 \texttt{baseline\_raw} 也独占 4 条，
说明当前没有任何单一方法能够覆盖全部问题类型。
40.6\% 的实例（203 条）对所有方法均构成失败，
这些实例代表了当前模型能力边界之外的任务，
进一步压缩上下文或改善检索均无法突破这一上限。

综合以上发现，本文建议：在实际部署场景中，优先选择技能记忆注入或文件级 RAG 作为成本优化手段；
在研究层面，将早停机制（针对瓶颈 C）与模型能力提升（针对难实例）列为下一步核心方向；
对于追求极致 token 节省的场景，需审慎评估 LLMlingua 等通用压缩方法
在具体仓库和问题类型上的适配性，避免因信息损失导致解决率下降反而提高 CPR。
