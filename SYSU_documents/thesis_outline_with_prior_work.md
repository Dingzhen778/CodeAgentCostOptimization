## 一、先说结论：当前两篇论文应该怎么融入

你现在最值得化用的两篇工作，不是并列挂在毕设里，而是扮演不同角色：

### 论文 A：`How to Compress Long-Context Code? An Exploratory Study`

这篇论文最适合提供：

- 上下文压缩方法分类与已有方法梳理

它在毕设中的定位应是：

- `相关工作基础`

它不应该直接成为毕设的“主方法章”，因为当前毕设主线不是“研究压缩算法本身”，而是“研究代码修复智能体的成本结构及优化方向”。

### 论文 B：`OmniGIRL: A Multilingual and Multimodal Benchmark for GitHub Issue Resolution`

这篇论文最适合提供：

- GitHub issue resolution 任务的更广阔背景
- issue resolution benchmark 的任务形态与评测视角

- “从单文件代码任务走向真实 issue 解决”的任务演进叙事

它在毕设中的定位应是：

- `任务背景与评测场景基础`

它不适合成为毕设的独立方法主体，因为你的毕设主线不是“提出新 benchmark”，而是“分析与优化现有 agent 的推理成本”。

## 二、旧 Web3 毕设是怎么“完美融入已发表论文”的

旧 Web3 毕设的组织逻辑大致是：

1. 先用“生态调研”建立问题背景和研究对象。
2. 再用“漏洞发现与开发者态度分析”建立问题分类与研究动机。
3. 再把这些前期成果上升为“检测方案”。
4. 然后落实为“工具设计与实现”。
5. 最后做实验评估。

它的高明之处在于：

- 已发表成果没有被孤立陈列，而是被转化成后续章节的前提。
- “调研结果”服务于“问题定义”。
- “问题定义”服务于“方法设计”。
- “方法设计”再服务于“系统实现和实验验证”。

对你当前毕设来说，应该照搬的不是 Web3 题材，而是这种组织方法：

- 把已有论文先变成“研究基础”
- 再把研究基础串成一条新的总问题
- 再让后续平台、分析和优化实验接上来

## 三、建议的毕设总主线

建议把毕设统一为下面这条主线：

> 随着代码智能体从代码补全走向真实 GitHub issue resolution，任务上下文显著增长，仓库级修复过程中的推理成本迅速上升。已有研究表明，长上下文压缩在代码任务中存在明显质量损失风险；而 issue resolution benchmark 的发展进一步说明，真实修复任务比传统代码生成更复杂。基于此，本文聚焦代码修复智能体在 `SWE-bench` 场景中的高 token 成本问题，构建实验闭环，分析成本结构，并提出面向代码理解阶段的上下文压缩与成本优化思路。

这条主线的优点是：

- 能把你的两篇论文都吸收进去
- 又不会偏离当前毕设摘要
- 并且能自然引出当前实验平台和方法比较

## 六、不推荐的融入方式

### 1. 不建议把两篇论文机械拆成独立章节

原因：

- 容易导致主线断裂
- 会让毕设像论文合集而非统一研究
- 其中一篇偏 benchmark，一篇偏长上下文压缩，都不是当前毕设的最终主问题

### 2. 不建议强行把 OmniGIRL 写成本文的核心实验基准

原因：

- 当前毕设实验主体明显是 `SWE-bench + mini-swe-agent`
- OmniGIRL 更适合作为任务背景与 benchmark 生态补充

### 3. 不建议把长上下文压缩论文的实验结果原样当作当前毕设结论

原因：

- 那篇论文主要面向长上下文代码任务的压缩研究
- 当前毕设聚焦 repair agent 的过程性成本
- 两者相关，但不完全同题
