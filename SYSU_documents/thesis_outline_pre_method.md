适用范围：本大纲仅覆盖“方法”和“实验”之前的内容，用于先搭建论文前半部分的叙事框架。

写作原则：

- 不提前写实验结果、实验数字和方法优劣排序。
- 不把当前观察写成确定性结论，除非已经进入实验章节。
- 前半部分的目标是提出问题、定义问题、说明研究空白、建立研究动机。
- 与当前项目无关的旧模板内容不应继续沿用。

## 一、绪论

### 1.1 研究背景与意义

建议覆盖的内容：

- 大语言模型在软件工程中的应用快速发展，代码智能体已进入缺陷定位、程序修复、测试生成、代码理解等任务。
- 自动程序修复是代码智能体的重要应用场景，且真实仓库级修复任务比单文件代码生成更复杂。
- 当前代码修复智能体往往依赖多轮交互、代码仓库搜索、文件阅读和环境验证，推理成本高。
- 高 token 消耗会直接影响运行成本、响应效率、可扩展性和真实部署价值。
- 因此，有必要将“成本”作为独立研究问题进行系统分析，而不是只关注成功率。

### 1.2 研究问题提出

建议明确的问题：

- 代码修复智能体的推理成本主要来自哪些环节。
- 仓库级修复为什么比普通代码生成更容易产生上下文膨胀。
- 现有上下文压缩与检索方法是否真正适配代码修复智能体。
- 如何在不显著损害修复能力的前提下降低推理成本。

### 1.3 研究目标与研究内容

建议写法：

- 构建面向代码修复智能体的实验与分析闭环。
- 从任务过程而非单轮输入角度分析推理成本结构。
- 面向仓库级修复任务提出上下文压缩与成本优化思路。
- 为后续方法设计、实验评测与部署讨论提供依据。

### 1.4 论文整体结构

建议简述各章安排，不展开细节，不夹带实验结论。

## 二、相关背景

### 2.1 大语言模型与代码生成

建议覆盖的内容：

- 大语言模型在自然语言和代码任务中的能力演进。
- 代码生成、代码补全、代码理解与软件工程任务的关系。
- 大模型在复杂代码任务中的局限，如上下文窗口、幻觉、长链路推理成本等。

### 2.2 代码智能体的基本工作机制

建议覆盖的内容：

- 代码智能体区别于单轮代码生成模型的核心特点。
- 智能体通常具备搜索、读取、编辑、执行、验证等工具调用能力。
- 多轮交互式决策如何带来更强问题求解能力，也带来更高过程性成本。

### 2.3 自动程序修复与仓库级修复任务

建议覆盖的内容：

- 自动程序修复的基本目标与典型流程。
- 仓库级修复任务需要理解 issue、关联代码、测试反馈和环境约束。
- 仓库级任务的困难不只是补丁生成，还包括代码定位与上下文组织。

### 2.4 SWE-bench 与代码修复评测场景

建议覆盖的内容：

- `SWE-bench` 的任务形式、输入输出和评测意义。
- 为什么 `SWE-bench` 能较好反映真实软件工程修复场景。
- 该基准为何适合用来研究代码修复智能体的成本问题。

### 2.5 token、上下文窗口与推理成本

建议覆盖的内容：

- token 是模型输入输出成本的直接计量单位。
- 长上下文不仅增加单步成本，也会在多轮交互中持续累积。
- 对代码智能体而言，成本应被理解为过程性成本，而不只是最终 prompt 长度。

## 三、相关工作

### 3.1 基于大语言模型的自动程序修复

建议写作重点：

- 回顾大语言模型在 APR 中的应用路径。
- 区分单轮补丁生成与多轮交互式修复。
- 强调现有工作多关注修复成功率、补丁正确性或基准表现。

### 3.2 代码智能体与软件工程智能体

建议写作重点：

- 综述能够调用工具、操作仓库环境的软件工程智能体。
- 介绍这类工作如何把问题求解流程从“直接生成”扩展为“搜索-理解-修改-验证”。
- 说明现有智能体研究通常强调能力提升，而对成本结构关注不足。

### 3.3 仓库级代码理解与检索增强

建议写作重点：

- 介绍 repository-level 代码任务中的检索增强思路。
- 比较文件级检索、片段级检索、函数级裁剪等路径。
- 指出现有检索工作主要提升相关性与准确性，不一定显式优化 agent 过程成本。

### 3.4 长上下文处理与上下文压缩

建议写作重点：

- 回顾长上下文压缩、重排序、摘要、裁剪等方法。
- 区分通用文本压缩与代码场景压缩。
- 说明代码任务中的结构信息、依赖关系与跨文件语义使压缩更复杂。

### 3.5 成本优化、效率优化与成功率权衡

建议写作重点：

- 综述已有关于推理成本、时延、工具调用效率、预算控制的相关研究。
- 强调成本与成功率之间可能存在权衡关系。
- 指出现有研究少有针对代码修复智能体进行阶段化成本归因。

### 3.6 现有研究不足总结

建议归纳为几类空白：

- 关注“能否修复”多，关注“以什么成本修复”少。
- 关注最终输入压缩多，关注多轮执行过程成本少。
- 关注检索效果多，关注历史 observation 冗余与无效循环少。
- 缺少面向真实仓库级修复场景的系统成本分析框架。

## 四、问题定义与研究动机

### 4.1 代码修复智能体任务流程抽象

建议内容：

- 将代码修复过程抽象为搜索、阅读、编辑、验证、提交等阶段。
- 说明这些阶段共同构成完整的任务执行链路。
- 强调成本来自整个过程，而不是单次补丁生成。

### 4.2 推理成本的构成与表现形式

建议内容：

- 输入代码上下文、历史 observation、工具执行反馈和模型输出都会带来 token 消耗。
- 成本既包括单步输入长度，也包括多轮累积成本。
- 在实际部署中，还应考虑运行时延和单位成功成本等扩展维度。

### 4.3 仓库级场景下高成本问题的来源分析

这里只写研究动机，不写实验结论。建议写成：

- 代码仓库搜索空间大，容易产生过宽候选范围。
- 智能体常需反复读取多个文件以建立跨文件理解。
- 多轮交互中历史 observation 容易累积，导致后续轮次输入不断膨胀。
- 若缺少有效预算控制，失败样本可能在后期出现低收益重复操作。

### 4.4 本文关注的问题边界

建议明确：

- 本文研究对象是代码修复智能体，而非通用聊天场景或纯代码补全任务。
- 本文关注的是真实仓库级修复过程中的推理成本问题。
- 本文讨论的优化对象主要是上下文组织、检索范围、历史信息管理与交互效率。
- 本文不把模型预训练、基础模型架构修改作为主要研究对象。

### 4.5 本文的研究切入点

建议写法：

- 从“过程性成本分析”切入，而非仅做总 token 统计。
- 从“代码理解阶段优化”切入，而非只压缩最终 prompt。
- 从“适配代码修复任务结构”的方法设计切入，而非直接复用通用文本压缩结论。

## 五、研究总体框架

这一章可作为前置章节，也可部分并入后续方法章。若保留，建议只写总览，不展开具体实验结果。

### 5.1 总体研究思路

- 以 `mini-swe-agent` 为执行主体。
- 以 `SWE-bench` 为主要任务场景。
- 从实验闭环搭建、成本分析、方法设计三个层面推进研究。

### 5.2 实验与分析对象概述

- 真实 issue 驱动的自动补丁生成任务。
- 代码仓库环境、执行轨迹、补丁结果与 token 日志共同构成分析对象。

### 5.3 成本分析框架概述

- 对智能体运行轨迹做阶段化划分。
- 从阶段、轮次和样本层面分析成本分布。
- 为后续方法设计提供问题定位依据。

### 5.4 上下文压缩方向概述

- 检索缩减。
- 历史 observation 压缩。
- 无效循环控制。
