# 论文主要内容及创新点概述

本文面向代码修复智能体在真实补丁生成任务中的高 token 消耗问题，围绕上下文压缩与成本优化展开研究。依托 `mini-swe-agent` 与 `SWE-bench`，本文构建了包含 Docker 运行、patch 生成、trajectory 记录和 token 统计的完整实验链路，并对搜索、阅读、编辑、验证、提交等阶段进行成本分析。研究表明，主要成本来自代码理解阶段，尤其是文件阅读与历史 observation 累积，而非补丁编辑本身。基于此，本文提出以代码理解阶段为核心的成本优化思路，重点从检索缩减、历史 observation 压缩和无效循环控制三个方向降低无关上下文消耗。创新主要体现在：提出阶段化 token 消耗分析框架；提出面向代码理解阶段的上下文压缩思路；构建面向 `SWE-bench` 代码修复任务的成本优化研究闭环。

# 创新点 1：提出面向代码修复智能体的阶段化 token 消耗分析框架

针对现有研究多停留在总 token 或平均成本统计、难以解释真实开销来源的问题，本文提出阶段化 token 消耗分析框架。该框架将代码修复智能体执行过程划分为搜索、阅读、编辑、验证、提交等阶段，并结合真实 trajectory 与日志统计各阶段成本占比。研究发现，代码理解阶段而非补丁编辑阶段才是主要成本来源，其中 `read` 是最大单项开销。该框架为后续成本优化提供了明确依据。

# 创新点 2：提出以代码理解阶段为核心的上下文压缩与成本优化思路

本文基于真实实验发现，代码修复智能体的高成本主要来自代码搜索、代码阅读及历史上下文累积，而不是最终补丁编辑。据此，本文提出以代码理解阶段为核心的上下文压缩思路，将优化重点从简单压缩最终 prompt 转向“减少读什么”和“减少保留什么”。具体包括检索缩减、历史 observation 压缩和无效循环控制三个方向，使成本优化更加符合代码修复任务的真实结构与实际需求。

# 创新点 3：构建面向 SWE-bench 代码修复任务的成本优化研究闭环

本文构建了面向 `SWE-bench` 代码修复任务的成本优化研究闭环，实现了 issue 输入、Docker 运行、patch 生成、trajectory 记录和 token 统计的一体化实验平台。该闭环能够同时采集补丁结果、执行轨迹与成本数据，使研究不再只关注“是否修复成功”，还能够进一步分析“以怎样的成本完成修复”。这一平台为后续压缩方法的公平比较、实验复现与持续扩展提供了稳定基础。

# 自我评价

本人在硕士阶段围绕大语言模型驱动的软件工程智能体开展研究，重点聚焦代码修复智能体在真实补丁生成任务中的高成本问题。在论文工作中，我完成了从实验链路搭建、补丁生成、轨迹记录到 token 统计分析的完整实现，并在 `SWE-bench` 场景下对不同模型的运行过程进行了阶段化分析。通过研究，我掌握了代码智能体、自动程序修复、Docker 实验环境和评测基准等相关技术，也逐步形成了从实验现象中抽象问题、再从成本结构中提炼优化方向的研究思路。总体而言，本人具备较强的工程实现能力和一定的科研分析能力，但也认识到当前工作在实验规模、方法实现深度和统计充分性方面仍有不足，后续仍需进一步提升方法设计与论文凝练能力。

# 申请报告

本文围绕代码修复智能体在真实软件工程任务中的高 token 消耗问题展开研究，提出了面向代码修复任务的上下文压缩与成本优化思路。论文的新见解在于：代码修复智能体的主要成本并不集中在补丁编辑阶段，而主要来源于代码理解阶段，尤其是代码搜索、文件阅读以及历史上下文累积。论文的新方法在于：构建了面向 `SWE-bench` 的代码修复智能体实验闭环，提出了阶段化 token 消耗分析框架，并据此形成了以检索缩减、历史 observation 压缩和无效循环控制为核心的成本优化方向。其意义在于推动软件工程智能体从“能完成任务”进一步走向“以更低成本完成任务”。目前仍有待继续探讨的问题包括：扩大实验规模以验证结论稳定性，实现具体压缩方法并开展系统对比实验，以及进一步联合考察成功率、时延和单位成功成本等指标。
